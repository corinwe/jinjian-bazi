## 一、核心问题

老板明确指出：「我觉得这个机制还是不太靠谱。因为我们这一套要对外输出就必须一致性稳定。那你去网上看看有没有这样标准化、工程化、自动化的一些方法和机制。」

当前机制的问题：
- 依赖LLM「记住」加载skills → 对话长了就忘
- 依赖LLM「自觉」执行验证步骤 → 没有硬约束
- 依赖LLM「手动」计算身强弱/藏干 → 容易算错
- **LLM的输出是概率分布，不是确定性函数**

## 二、调研结果汇总

### 来源A：学术论文 — BaZi-Based Character Simulation Benchmark

**标题**：BaZi-Based Character Simulation Benchmark: Evaluating AI on Temporal and Persona Reasoning
**收录**：WordPlay Workshop 2025
**链接**：https://arxiv.org/abs/2510.23337

**核心架构**——四组件：
1. **Input Layer**：出生信息（birthday, gender, place of birth）
2. **BaZi Rule Analysis**：rule-based BaZi mapping（规则引擎）
3. **BaZi Reasoning**：Ten Gods / ShenSha / Pattern Structures（逻辑推理）
4. **Scenario Interpretation**：5 domains（wealth/health/career/kinship/relationships）

**关键数据**：
- 符号推理+LLM混合系统 → 比纯LLM高 **30.3%-62.6%** 准确率
- 输入错误的BaZi信息 → 准确率**暴跌20%-45%**（证明规则层在做真实工作）
- 5大域：wealth/health/career/kinship/relationships ← 和我们的12大板块高度一致（财富/健康/事业/亲情/关系）

**方法论创新**——三要素：
1. 最小输入 → 丰富输出（只需出生信息，生成全维度Persona）
2. 符号逻辑集成（规则BaZi映射 + 十神/神煞/格局逻辑推理）
3. 场景自适应（Persona不是静态的，随健康/财富/事业上下文动态变化）

**对我们的启示**：
- ✅ 规则引擎先行 → LLM后行 = 正确路径
- ✅ 5大域的生命事件分类概念可以引入我们的板块设计
- ✅ 30-62%的精度提升验证了混合架构的有效性

### 来源B：开源工程 — cantian-ai/bazi-mcp

**项目**：Bazi MCP Server by Cantian AI
**链接**：https://github.com/cantian-ai/bazi-mcp
**Stars**：~390 | **Forks**：157 | **License**：MIT

**定位**：针对GPT/DeepSeek等AI工具排盘不准确的问题，提供精准的八字数据

**功能**：
- 输入公历/农历时间 → 输出结构化JSON（八字四柱+藏干+十神+纳音+空亡+星运+自坐）
- 输入参数：solarDatetime / lunarDatetime / gender / eightCharProviderSect(早晚子时配置)
- 支持Streamable HTTP / Stdio两种传输方式
- 输出JSON的字段名是中文（"性别""阳历""八字""年柱"等）

**输出示例**：
```json

  "性别": "男",
  "八字": "戊寅 己未 己卯 辛未",
  "日主": "己",

    "地支": {"地支":"寅","五行":"木","阴阳":"阳",
```

**对我们的启示**：
- ✅ 结构化JSON输出是行业标准做法，我们的bazi-engine.py方向正确
- ✅ 字段设计（主气/中气/余气）和我们的100/60/30标准一致

### 来源C：生产级LLM结构化输出最佳实践

**文章**：Structured Output in Production: Getting LLMs to Return Reliable JSON
**作者**：Tian Pan
**链接**：https://tianpan.co/blog/2025-11-10-structured-output-production-llms

**三种结构化输出机制**：

| 机制 | 描述 | 失败模式 | 适用场景 |
|:---|:---|:---|:---|
| 1. Prompt-Only JSON | 简单要求模型输出JSON | 字段缺失、类型错误、enum值不准 | 探索性提取 |
| 2. Constrained Decoding | Token级别Schema约束 | 过于严格时语义质量下降 | 数据提取Agent工具调用 |
| 3. 混合方案 | 约束解码+验证层+重试 | ⋯ | 生产环境推荐 |

**生产级质量保障四层模型**（超出Schema验证等）：
1. **Schema验证层** — JSON结构正确性
2. **语义验证层** — 值在合理范围内、enum值正确
3. **回退策略** — 检测到错误时自动重试（带修正提示）
4. **监控告警** — 生产环境中持续跟踪质量指标

**对我们的启示**：
- ⚠️ 纯Prompt方式（我们之前用的）是最不可靠的机制
- ✅ 需要验证层（不仅仅是KV存储有值，还要检查值的合理性）
- ✅ Schema设计要够宽松以免过度约束导致语义下降

### 来源D：AI Agent输出质量保障工程实践

**文章**：AI Agent 输出质量保障工程实践：从「玄学调参」到「稳健性检验」
**作者**：wesley028
**链接**：https://juejin.cn/post/7624420198007865363

**三大翻车模式**（和我们完全一致）：

| 翻车模式 | 表现 | 根因 | 我们的命中情况 |
|:---------|:-----|:-----|:------------|
| ① 格式漂移 | 同一任务输出格式不统一 | output format在prompt中优先级被长上下文稀释 | ✅ 报告有时12板块有时少几块 |
| ② 上下文遗忘 | 长对话后违反初始规则 | LLM注意力机制被近期对话主导 | ✅ 对话长了忘加载skills |
| ③ 指令注入 | Agent指令被用户干预 | 指令在可读上下文中暴露 | — |

**四维检验框架**：

| 维度 | 方法 | 判定标准 |
|:-----|:-----|:--------|
| 一致性测试 | 同一输入运行N≥5次 | 格式一致率≥98%→PASS，<95%→FAIL |
| 边界测试 | 极端/空/恶意输入 | 有标准的处理方式 |
| 稳定性测试 | 连续长时间运行 | 质量不随运行时间下降 |
| 错误恢复测试 | 故意给出错误输入 | Agent能检测并恢复 |

**对我们的启示**：
- ✅ 一致性测试（N≥5）可以自动化纳入验证脚本
- ✅ 我们遇到的「报告少板块」「时序错误」「称谓矛盾」都是行业常见问题
- ✅ 格式一致率≥98%可以作为我们的验收标准

## 四、关键认知

> LLM的输出是概率分布，不是确定性函数。  
> 规则引擎的输出是确定的 → 用它做骨架  
> LLM的输出是概率的 → 用它在骨架上做推理  
> 验证层的输出是二元的 → 用它检查结果  
> 三者各司其职，互不替代，这是行业共识。
