# 金鉴体系：AI专业输出的物理约束架构

> **核心命题：** 大模型的输出是概率分布，不是确定性函数。对于需要高精度的专业分析场景，如何将概率输出"锁死"在可信轨道上？

## 一、问题本质

LLM有三个结构性缺陷：
- **爱猜数字**：问数值第一反应是"猜"而不是"查"（OpenAI 2025论文证实）
- **格式漂移**：同一个模板输出100次有97种不同格式
- **上下文遗忘**：对话长了早期约束被稀释

结论：不能靠AI"记住"规则，必须用物理机制代替自觉。

## 二、五层物理锁死架构（L0→L4）

### L0 人格宪法层 ⭐⭐⭐⭐⭐
写入系统提示词底部的铁律，每轮对话自动加载，无法删除或修改。
- 10条核心指令 + 5步标准工作流 + 违规后果
- **物理强度：** 每轮加载，无法绕过

### L1 系统配置层 ⭐⭐⭐⭐
通过Agent配置文件实现三层约束：
- 审批模式（低风险自动过/高风险需确认）
- 命令白名单（核心流程自动放行）
- 文件系统权限（可读写路径白名单）
- 操作钩子（每次工具调用自动触发校验）

### L2 项目规则层 ⭐⭐⭐
工作目录下写入规则文件，Agent在该目录下自动加载。
- 数据要素定义 + 标准输出格式（含签名要求）
- 输出前自检清单 + 质量标准

### L3 Pipeline主控层 ⭐⭐⭐⭐⭐
唯一入口——不跑Pipeline拿不到骨架数据和context文件。
- 排盘验证 → 标准骨架 → 子任务分发 → 回写校验 → 全量验证 → 推库
- 每个环节被后置脚本锁死，不可手动跳过

### L4 验证脚本层 ⭐⭐⭐⭐
多层自动验证门禁：
- 格式验证 → 数据一致性验证 → 年份逻辑验证 → 全面审计
- 提交前Hook（git commit时自动触发，不合格拒绝提交）
- 每层独立校验，层层封锁

## 三、四大设计原则

1. **确定性计算代码化，LLM只做推理**
   代码引擎输出所有数字 → LLM在确定性数据上做语义分析 → 验证层自动校验。LLM不做任何数学计算。

2. **数据源头盔：先查再说**
   任何时候输出数字，强制流程：查数据源 → 得数字 → 验一致性 → 输出。不查数据源直接说数字 = 不可信。

3. **多层叠防，冗余校验**
   任何单层可被绕过，但多层叠防使完全绕过成本极高。每层独立校验，一处不通所有输出被阻断。

4. **物理约束 > 自觉**
   所有约束必须是代码/脚本/配置文件，不能是「记住要做X」。规则写入而非记住，脚本执行而非手动，拦截自动而非事后。

## 四、为什么有效

| 传统做法 | 问题 | 本架构方案 |
|:---------|:-----|:-----------|
| 让AI记住规则 | 对话长了就忘 | 规则写入配置+提示词，每轮自动加载 |
| 手动验证 | 容易漏 | 自动验证脚本，commit前拦截 |
| AI参与计算 | 概率输出不稳定 | 确定性计算代码化，LLM只做推理 |
| 单点依赖 | 一点突破全盘输 | 多层叠防，每层独立校验 |

## 五、一句总结

**不是让AI变得更可靠，而是让不可靠的AI跑在可靠的轨道上。**
