# 会话记忆管理（Session Memory Management）

## 概述

Agent 的记忆系统分四层，层层递进。本 skill 定义各层的使用时机、决策树和禁忌，确保：
- 用户不需要重复交代背景
- Agent 主动回溯历史，不让用户说第二遍
- 踩过的坑自动沉淀，下次不重犯

---

## 四层架构 + STATE.md 物理化持久

```
STATE.md 文件 (物理化持久 — 模型忘记但文件记得)
    ↓ 每次会话自动读取，每轮更新
持久记忆 (memory — 4000字符上限)
    ↓ 永久存储，跨所有会话
历史检索 (session_search — 按需回溯)
    ↓ 回溯过去对话，按需查询
技能库 (skill_manage → skills)
    ↓ 可复用工作流，自动加载+自动修补
当前上下文
    ↓ 单会话内，结束即清
```

### ⚠️ 核心原则（来自Loop Engineering）

> **模型会忘记，但文件不会。**
> 六六"隔天就忘"的根因：依赖memory()的4000字符上限 + 跨会话丢失。
> 解决方案：多一层物理化持久层——STATE.md文件。

### STATE.md 使用规则

每个Profile的项目根目录或home目录下必须有 `STATE.md`，作为**不可丢失的跨会话状态锚点**。

#### 什么时候需要STATE.md
- 项目有多个长期任务（如在途票追踪、成绩追踪、产品开发）
- 用户投诉"你又忘了"
- 任务需要跨多个会话持续跟踪
- memory()已接近4000字符上限

#### STATE.md 模板

```markdown
# [项目/Profile名] 状态文件

> 最后更新：YYYY-MM-DD HH:mm
> 核心信条：模型不记得，但文件记得。每次会话开始自动读取。

## 当前活跃任务
| # | 任务 | 优先级 | 状态 | 上次行动 |
|---|------|--------|------|---------|

## 待处理事项
...

## 尝试过但失败的方案
...

## 下一步待尝试
...
```

#### 会话生命周期中的STATE.md

```
会话开始 → read_file STATE.md → 恢复上下文
     ↓
正常执行任务
     ↓
会话结束 → cron/Agent 更新 STATE.md → 状态持久化
     ↓
下一会话开始 → read_file STATE.md → 无缝续接
```

#### 什么时候写STATE.md
- 完成任务/子任务后 — 更新状态 + 记录结果
- 尝试方案失败后 — 记录失败原因 + 替代方案（避免重复踩坑）
- 发现新信息后 — 更新关键上下文
- 会话结束时 — 确保所有状态已持久化

#### 什么时候不写STATE.md
- 仅查询/只读操作（问个简单问题就走）
- 临时讨论（与项目长期任务无关）

## 四层架构（原版）

```
持久记忆 (memory)
    ↓ 永久存储，跨所有会话

    ↓ 回溯过去对话，按需查询
技能库 (skill_manage → skills)
    ↓ 可复用工作流，自动加载+自动修补
当前上下文
    ↓ 单会话内，结束即清
```

## 层1：持久记忆（memory）

### 什么时候存 ✅
- 用户告知的**个人偏好**（称呼、风格、时区）
- **环境事实**（OS、项目根目录、API 密钥位置）
- **项目约定**（命名规范、架构决策、技术栈）
- 用户**纠正你**的事情
- 用户明确说"记住这个"
- 你发现的、未来会话一定会用到的**稳定事实**

### 什么时候不存 ❌
- 任务进度和临时状态（PR 编号、commit SHA、bug 修复记录）
- 7 天内会过时的信息
- 当前会话的中间产物
- 工具调用细节（如 "submitted PR #42"）

### 存储规范
- **声明式陈述句**：`用户偏好简洁回答` ✓ | `必须简洁回答` ✗
- **紧凑优先**：每条只存一个事实，控制在 1-2 句话
- **归类**：`memory` 目标存环境/项目/技术细节，`user` 目标存用户身份/偏好

## 层2：历史检索（session_search）

### 主动搜索的信号
用户说以下关键词时，**必须先搜，不问让用户重复**：
- "上次那个……"、"之前……"、"我记得……"
- "我们讨论过……"、"这个之前说过"
- 提及曾做过的功能、修复过的 bug、讨论过的方案

### 搜索技巧
- 用 `OR` 连接关键词扩大召回（`数据库 OR 方案 OR 架构`）
- 默认 AND 会缩小范围，先 OR 后 AND
- 搜不到时换同义词再试一次

### 浏览最近会话
- 无参调用 `session_search()` 查看近期活动，零成本
- 适合用户说"最近我们在做什么"

---

## 层3：技能库（skill_manage）

### 什么时候创建 skill
- 完成复杂任务（5+ 工具调用）
- 克服了棘手的错误
- 发现可复用的工作流
- 用户要求"把这个流程记下来"

### 什么时候更新 skill
- 加载 skill 后发现指令过时/有坑 → **立即修补，不等用户指出**
- 新发现已知流程的注意事项
- OS/环境变更导致旧步骤失效

---

## 层4：当前上下文

- 当前会话内的全部工貝调用、文件读写、对话历史
- 用 `todo()` 追踪当前任务列表
- 任务完成即清理
- **会话结束 → 上下文消失**（这是正常的，由前 3 层覆盖）

---

## 决策树

```
用户说了某个东西
├─ 是在当前会话提过的？
│   └─ ✅ 直接接上下文，不用搜
├─ 看着像在提旧事（"上次"、"之前"）？
│   └─ ✅ 先 session_search，不问用户
├─ 完全新的内容？
│   └─ ✅ 正常处理，不额外动作
└─ 你回答完了发现遗漏了已知信息？
    └─ ✅ session_search 补上，承认遗漏
```

---

## 零重复错误原则

每次新会话/任务开始时：
1. **读 ERROR_LOG.md**（如果项目有这么个文件）
2. **读 TASK_BOARD.md**（如果项目有）
3. **检查 memory** 中的相关约束
4. **session_search** 确认有没有历史相关的坑

## 跨 Agent 复用说明

本 skill 设计为**平台无关**，任何 Hermes Agent 都可加载。
加载后 Agent 会自动获得会话记忆管理的最佳实践，无需修改系统提示词。

### 加载方式
- 自动扫描匹配（如用户问"还记得上次那个吗？"时自动加载）
- 或显式加载：`skill_view(name='session-memory-management')`

### 预期效果
加载后的 Agent 应该：
- 主动搜历史，不問用户重复
- 正确使用 memory 存/不存
- 完成复杂任务后自动沉淀为 skill
- 每次开始先检查 ERROR_LOG（本项目的错误日志路径）
