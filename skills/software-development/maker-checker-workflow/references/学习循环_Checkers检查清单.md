# 学习循环 · Checker 检查清单

> 来源：2026-07-02 金鉴真人·09号视频 + 素材11 学习循环实战教训
> 用途：当收到新知识（截图/视频/素材/口述修正）时，用此清单做 Checker 验证
> 对应 skill: `maker-checker-workflow` 第五步（学习循环）

## 一、变更清单完整性检查

### 1.1 规则覆盖率

逐条检查变更清单中的规则：

```
□ 每一条新规则是否有具体内容（非"待填充"占位符）？
□ 每一条新规则是否标注了来源？
   ├─ 素材行号（如素材11行349~433）
   ├─ 视频时间戳（如09号视频第1:26:55）
   ├─ 截图位置（如09号视频截图 §1:58）
   └─ 实战校准（如老板口述修正）
□ 修正规则是否标注了"旧→新"对比？
□ 删除规则是否说明了删除原因？
```

### 1.2 章节完整性

```
□ 是否整个§章节新增？→ 确认所有小节都已覆盖
□ 是否§章节修正？→ 确认修正部分与不修正部分有明确分界
□ 是否已有内容与新规则冲突？→ 检查是否存在矛盾
□ 版本号是否正确递增？→ v3.9 → v4.0 ✓
```

## 二、Skill 更新检查

### 2.1 新增内容

```
□ 新增的§章节编号与现有体系不重复
□ 新增内容中有引用到已有§时，引用§号正确
□ 关键字/口诀与实际规则一致
```

### 2.2 修正内容

```
□ 修正后的规则与原始素材一致
□ 修正说明标注了"vX.X核心变更"或类似标记
□ 所有引用该规则的其他§也已同步更新
```

## 三、报告影响检查

### 3.1 受影响报告全覆盖

```
□ 循环引擎输出的 affected_reports 是否完整？
   └─ 如果不完整，手动扫描补充
□ 每个受影响报告确认以下字段已更新：
   ├─ §👑 能量链条（如适用）
   ├─ §7 发财方式（如适用）
   ├─ §16 合作伙伴（如适用）
   └─ §17 破财条件（如适用）
□ 标准格式报告（非专项报告）是否也需要更新？
```

### 3.2 数据一致性

```
□ 同一数据在不同报告中是否一致？
   └─ 同一命主的财星分在四份报告中必须相同
□ 更新后的数据与引擎JSON是否一致？
□ §1总览表与§8详细分析的财星分是否一致？
```

## 四、推库检查

```
□ 所有变更已 git add
□ commit message 格式正确
□ git push 成功
□ 远程仓库验证：git log --oneline -1 确认最新commit
```

## 五、反模式截停

| 检查项 | PASS/FAIL |
|:-------|:----------|
| 变更清单中有"待填充"占位符？ | FAIL → 必须填充完整 |
| 规则没有来源标注？ | FAIL → 每规则必须有来源 |
| 只改了skill没改报告？ | FAIL → 报告也必须更新 |
| 只改了部分报告漏了某份？ | FAIL → 全量覆盖 |
| 没有推库？ | FAIL → 必须先推库 |

## 口诀

```
学习循环三阶段，Learm Verify Apply
变更清单先完整，规则内容不能空
Skill更新完不算完，报告覆盖要全看
推库之前再检查，数据一致才能过
```
