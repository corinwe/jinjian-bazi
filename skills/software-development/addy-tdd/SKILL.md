# Test-Driven Development (addyosmani 生产级TDD)

## 概述

在写代码之前先写一个会失败的测试。对于bug修复，在尝试修复之前先用测试复现bug。测试就是证据——"看起来没问题"不算完成。

## TDD循环

```
    RED                GREEN              REFACTOR
 写一个会失败的测试 → 写最少的代码让它通过 → 清理实现 → (重复)
      │                    │                    │
      ▼                    ▼                    ▼
   测试失败             测试通过             测试仍然通过
```

### Step 1: RED — 写一个会失败的测试
先写测试。它必须失败。一个立即通过的测试什么都没证明。

### Step 2: GREEN — 让它通过
写最少的代码让测试通过。不要过度设计。

### Step 3: REFACTOR — 清理
测试通过后，在不改变行为的前提下改进代码：
- 提取共享逻辑
- 改进命名
- 消除重复
- 必要时优化

每步重构后运行测试确认没破坏任何东西。

## Prove-It模式（Bug修复）

当有bug报告时，**不要先尝试修复。** 先写一个复现bug的测试。

```
Bug报告到达
       │
       ▼
  写一个展示bug的测试
       │
       ▼
  测试失败（确认bug存在）
       │
       ▼
  实现修复
       │
       ▼
  测试通过（证明修复有效）
       │
       ▼
  运行完整测试套件（无回归）
```

## 测试金字塔

```
          ╱╲
         ╱  ╲         E2E测试 (~5%)
        ╱    ╲        完整用户流程，真实浏览器
       ╱──────╲
      ╱        ╲      集成测试 (~15%)
     ╱          ╲     组件交互，API边界
    ╱────────────╲
   ╱              ╲   单元测试 (~80%)
  ╱                ╲  纯逻辑，隔离，毫秒级
╱──────────────────╲
```

**Beyonce法则：** 如果你喜欢它，你应该为它写测试。基础设施变更、重构和迁移不应该负责抓你的bug——你的测试负责。如果一次变更破坏了代码而你没有测试，那是你的问题。

### 测试规模

| 规模 | 约束 | 速度 | 示例 |
|------|------|------|------|
| **Small** | 单进程，无I/O，无网络，无数据库 | 毫秒 | 纯函数测试，数据转换 |
| **Medium** | 多进程OK，仅localhost，无外部服务 | 秒 | API测试+测试DB，组件测试 |
| **Large** | 多机器OK，允许外部服务 | 分钟 | E2E测试，性能基准测试 |

## 写好测试

### 测试状态，而非交互
断言操作的**结果**，而不是内部调用了哪些方法。验证方法调用序列的测试在重构时会破坏，即使行为不变。

### DAMP优于DRY
在测试中，**DAMP（描述性和有意义的短语）**更好。一个测试应该像一个规范一样阅读——每个测试应该讲一个完整的故事，不需要读者追踪共享辅助函数。

### 测试命名模式
```
[被测方法]_[场景]_[预期行为]

it('creates a task with title and default status', async () => { ... })
it('returns 400 when title is empty', async () => { ... })
```

### 测试结构：AAA模式
```
// Arrange - 安排
const input = { title: 'Buy groceries' };

// Act - 执行
const task = await taskService.createTask(input);

// Assert - 断言
expect(task.title).toBe('Buy groceries');
```

## 浏览器测试
- 使用真实的浏览器环境（Playwright/Cypress）
- 关键用户流程使用E2E
- 隔离UI组件使用组件测试
- 不要模拟浏览器API——使用真实的浏览器

## 反合理化（Anti-Rationalization）

| 借口 | 反驳 |
|------|------|
| "这个太简单不需要测试" | 简单的代码也会出bug，而且正好因为你觉得简单而忽略测试 |
| "我会稍后加测试" | 你不会。每次说这话都有证据 |
| "测试很难写" | 如果测试很难写，说明设计有问题 |
| "用户不会这么做" | 用户总会做你没想到的事 |
| "我没时间写测试" | 你没时间花双倍时间调试吗？ |

## 验证标准
- ✅ RED步骤：测试确实失败
- ✅ GREEN步骤：测试通过
- ✅ REFACTOR后：所有测试仍通过
- ✅ 完整测试套件通过
- ✅ 没有跳过或注释掉的测试
