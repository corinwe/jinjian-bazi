# Agents Best Practices — Harness设计最佳实践

> 供应商中立的Agent架构设计技能
> 来源：github.com/DenisSergeevitch/agents-best-practices

## 核心立场

Agent Harness是模型周围的控制平面。模型提议行动；Harness验证、授权、执行、记录、总结并返回观察结果。

### 默认架构

```
用户/任务
  → 指令和上下文构建器
  → 模型调用
  → 工具/行动提议
  → Schema验证
  → 权限决策
  → 执行或审批暂停
  → 结构化观察
  → 上下文更新
  → 在预算内重复或完成
```

## 关键设计原则

### 反馈循环原则
- **失败必须反馈到Harness**。重复的失败应该成为Harness功能（验证器、工具、文档、评估或策略），而不是重复提示建议
- **信任但验证（Trust but Verify）**。始终验证Agent的输出
- **粒度预算**。限制每次调用的预算（总步骤、总token、总时间）

### 工具设计
- 工具设计即Agent UX。命名、描述、参数和错误表面都是Agent的UI
- 暴露足够让Agent工作的最小工具集。太多选择等于没有选择
- 工具应该清晰地成功或失败，而不是静默返回空结果

### 权限模型
- 批准流：每次执行前require approval → 审批/修改/拒绝
- 认证上下文：工具自动附加身份信息，Agent不需要手动传递
- 权限随时间扩展：从保守开始，随信任增加而扩展

### 系统提示设计
- 清晰最优先，长度次之。清晰的指令比短的指令更有效
- 结构化System Prompt（章节标题、列表）
- 在提示中嵌入验证标准

## 运行时约束

| 约束 | 实现方式 |
|------|---------|
| 总步骤限制 | 循环计数 |
| 总token预算 | 跟踪prompt+completion |
| 工具执行超时 | 每个工具调用超时 |
| 上下文窗口管理 | 压缩/摘要/窗口化 |
| 幂等性 | 工具执行应该是幂等的 |

## 上下文管理

- **上下文腐烂（Context Rot）** 是Agent可靠性的最大敌人
- 渐进式上下文：先摘要→再按需加载细节
- 文件系统作为上下文载体（自文档化、可审计）
- 会话间的记忆：写入文件系统比结构化记忆更可靠

## 验证策略

### 分层验证
1. **Schema验证** - 工具输入/输出的类型检查
2. **确定性验证** - 测试、lint、结构化分析（计算型）
3. **Agent作为验证者** - 另一个Agent或同一Agent审查输出（推理型）
4. **人类审查** - 关键决策点

### 自我纠正
Harness应该内置自我纠正能力：
```
检测到失败 → 记录失败模式 → 限制/禁止该模式 → 自我纠正 → 重试
```

## 可观测性

- 结构化日志（JSON格式，包含traceId）
- RED指标：Rate/Errors/Duration
- 追踪：完整的调用链
- 审查轨迹：每次Agent行动都记录了"谁在何时做了什么"

## 检查清单

Harness审计时检查：
- [ ] 有前馈指南（规则/文档/提示）吗？
- [ ] 有反馈传感器（测试/验证/审查）吗？
- [ ] 工具有清晰的命名和错误表面吗？
- [ ] 权限控制到位吗？
- [ ] 上下文压缩策略吗？
- [ ] 有预算限制（步骤/token/时间）吗？
- [ ] 有自我纠正机制吗？
- [ ] 可观测性到位（日志/指标/追踪）吗？
- [ ] 人类在环机制吗？
- [ ] Agent能自我评价自己的工作吗？（执行者和评估者分离）
