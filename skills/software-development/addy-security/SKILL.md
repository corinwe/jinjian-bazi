# Security and Hardening (生产级安全加固)

## 概述

安全是每个开发者的责任。每个变更必须考虑安全影响。三层边界系统——数据进入输出之间每一步都要验证。

## OWASP Top 10防护

### 1. 注入（SQL、NoSQL、OS、LDAP）
- 始终使用参数化查询——绝不拼接字符串
- 使用ORM的内置转义
- 验证和清理所有用户输入
- 使用最小权限原则

### 2. 认证失效
- 使用标准的认证库（NextAuth、Auth0、Supabase Auth）
- 安全存储密码（bcrypt/argon2，绝不用明文）
- 实施速率限制
- 使用HTTP-only、Secure、SameSite cookies
- 实施MFA用于敏感操作

### 3. 敏感数据暴露
- 绝不硬编码密钥——使用环境变量
- 传输中加密（HTTPS/TLS）
- 静止时加密（数据库加密）
- 不要记录敏感数据

### 4. XML外部实体（XXE）
- 禁用XML外部实体解析
- 尽可能偏好JSON而非XML

### 5. 访问控制失效
- 在每个端点检查授权
- 使用最小权限原则
- 实施功能级访问控制
- 防止IDOR（不信任用户提供的ID）

## 密钥管理

```
绝不提交密钥到版本控制
密钥存储在环境变量或密钥管理服务中
开发/阶段/生产使用不同的密钥
密钥定期轮换
使用.env.local进行本地开发（已加入.gitignore）
```

## 三层边界系统

```
外部边界（用户/API输入）
    ↓ 验证和清理
内部边界（应用逻辑）
    ↓ 授权检查
数据边界（数据库/文件系统）
    ↓ 安全查询
输出边界（响应/渲染）
    ↓ 编码输出
```

## 安全清单

每次推送前检查：
- [ ] 用户输入是否已验证和清理？
- [ ] SQL查询是否参数化？
- [ ] 密钥是否从代码和日志中排除？
- [ ] 需要的地方是否检查了认证和授权？
- [ ] 输出是否编码以防止XSS？
- [ ] 依赖是否没有已知漏洞？
- [ ] HTTPS是否被强制使用？
- [ ] CSRF保护是否到位？
- [ ] 速率限制是否应用于API端点？
- [ ] 上传的文件是否经过验证？
