# Debugging and Error Recovery (生产级调试)

## 概述

五步故障排查：复现、定位、缩小、修复、防护。红线规则——如果修复花费的时间超过30分钟未解决，后退并重新评估。安全回退——当事情不确定时倾向于安全的默认值。

## 五步排查流程

```
Bug报告或测试失败
       │
       ▼
Step 1: 复现——可靠地复现问题
       │
       ▼
Step 2: 定位——找到问题的根因
       │
       ▼
Step 3: 缩小——创建最小复现
       │
       ▼
Step 4: 修复——实施修复并测试
       │
       ▼
Step 5: 防护——防止未来复发
```

### Step 1: 复现
- 精确的复现步骤是什么？
- 是确定性的还是间歇性的？
- 只在特定环境/数据下发生？
- 写一个失败的测试来证明bug的存在

### Step 2: 定位
- 使用二分法：注释掉一半的代码看问题是否消失
- 检查最新的变更（git bisect）
- 检查错误消息和堆栈跟踪
- 添加日志/断点
- 区分"表现"和"原因"

### Step 3: 缩小
- 创建最小复现（Minimal Reproduction）
- 消除不相关的代码和变量
- 确认最小复现确实展示了相同的问题
- 最小复现是修复的精确规范

### Step 4: 修复
- 修复应该只针对根因
- 不要修复症状
- 添加回归测试
- 运行完整的测试套件

### Step 5: 防护
- 是否存在类似的bug在其他地方？
- 类型系统能防止这类bug吗？
- 需要lint规则或静态分析检查吗？
- 文档是否应该更新？

## 红线规则

如果30分钟的主动排查没有产生进展：
1. 停止
2. 写下你已知的
3. 写下你假设的
4. 从新的角度重新开始

## 安全回退

当不确定一个修复的正确性时：
- 倾向于保守的修复
- 添加额外的日志来确认行为
- 使用功能标志控制风险代码
- 准备好回滚计划

## 常见陷阱

| 陷阱 | 更好的方法 |
|------|-----------|
| 直接修复症状 | 先找到根因 |
| 猜测而不是验证 | 用数据说话（日志/测试/堆栈跟踪） |
| 一次改变太多东西 | 一次改变一个变量 |
| 跳过写复现测试 | 复现测试是最精确的bug文档 |
| 认为"这不会影响其他" | 运行完整的测试套件 |
