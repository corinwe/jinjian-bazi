# Code Review and Quality (生产级代码审查)

## 概述

多维度代码审查与质量门禁。每次变更在合并前必须审查——没有例外。审查覆盖五个轴：正确性、可读性、架构、安全、性能。

**批准标准：** 当一次变更确定能改善整体代码健康状况时批准——即使它不完美。完美代码不存在——目标是持续改进。

## 五轴审查

### 1. 正确性
- 代码是否实现了它声称的功能？
- 是否匹配规格或任务要求？
- 边界情况是否处理（null、空值、边界值）？
- 错误路径是否处理（不仅仅是正常路径）？
- 是否通过所有测试？
- 是否存在off-by-one、竞态条件或状态不一致？

### 2. 可读性与简洁性
- 命名是否描述性且符合项目规范？
- 控制流是否直接？（避免嵌套三元运算符、深层回调）
- 代码是否逻辑组织？（相关代码分组、清晰的模块边界）
- 有没有应该简化的"聪明"技巧？
- 能否用更少的行数完成？（1000行代替100行是失败）
- 抽象是否值得它的复杂度？（不要在第三个用例之前泛化）
- 是否存在死代码？（无操作变量、向后兼容垫片、// removed注释）
- 是否有条件逻辑被硬塞到不相关的流程中？

### 3. 架构
- 变更是否符合系统设计？
- 是否遵循现有模式或引入了新模式？新模式是否有充分理由？
- 是否维护了清晰的模块边界？
- 是否存在应该共享的代码重复？
- 依赖流向是否正确？（无循环依赖）
- 重构是降低了复杂度还是只是转移了它？

### 4. 安全
- 用户输入是否经过验证和清理？
- 密钥是否从代码、日志和版本控制中排除？
- 需要的地方是否检查了认证/授权？
- SQL查询是否参数化？（无字符串拼接）
- 输出是否编码以防止XSS？
- 依赖是否来自可信来源且无已知漏洞？

### 5. 性能
- 是否有N+1查询模式？
- 是否有无限循环或不受限制的数据获取？
- 是否有应该是异步的同步操作？
- 是否有不必要的UI组件重新渲染？
- 列表端点是否缺少分页？
- 热路径中是否有大型对象创建？

## 变更规模控制

```
~100行变更   → 好。一次审查可以完成。
~300行变更   → 可接受（如果是单一逻辑变更）。
~1000行变更  → 太大。拆分它。
```

## 严重性标签

| 前缀 | 含义 | 作者行动 |
|------|------|---------|
| (无前缀) | 必须修改 | 合并前必须解决 |
| **Critical:** | 阻止合并 | 安全漏洞、数据丢失、功能破坏 |
| **Nit:** | 次要、可选 | 作者可以忽略 |
| **Optional:** | 建议 | 值得考虑但不必要 |
| **FYI** | 仅供参考 | 无需行动 |

## 批准标准

批准意味着：
- ✅ 代码达到或超过项目的质量标准
- ✅ 测试覆盖了变更
- ✅ 没有安全或性能问题
- ✅ 架构是合理的
- ✅ 整体代码健康状况得到改善

不批准意味着：
- ❌ 存在未解决的正确性问题
- ❌ 测试缺失或不足
- ❌ 存在安全漏洞
- ❌ 架构问题需要讨论
