# Spec-Driven Development (生产级规格驱动开发)

## 概述

在写任何代码之前，先写规格。PRD（产品需求文档）覆盖：目标、命令、结构、代码风格、测试和边界。规格是团队的单一事实来源。

## PRD结构

### 1. 目标和背景
- 为什么要做这个？
- 解决什么问题？
- 成功标准是什么？

### 2. 范围和边界
- 包括什么？
- 不包括什么（明确地说出来）？
- 依赖和前提条件是什么？

### 3. 命令和用法
用户/开发者如何与功能交互？
```bash
# CLI示例
npx my-tool build --output ./dist

# 或API示例
POST /api/v1/tasks
{ "title": "完成报告", "priority": "high" }
```

### 4. 实现结构
- 文件/模块组织
- 数据流
- 关键组件

### 5. 代码风格
- 命名约定
- 格式化规则
- 错误处理模式

### 6. 测试策略
- 测试什么？
- 如何测试？
- 边界情况是什么？

## 规格质量标准

| 标准 | 检查 |
|------|------|
| 完整性 | 规格覆盖所有需求 |
| 清晰 | 没有歧义——一人一个解释 |
| 可测试 | 规格中的每个断言都是可验证的 |
| 一致 | 规格内部没有矛盾 |
| 可行 | 规格中的内容在时间和预算内可实现 |

## 常见的规格陷阱

| 陷阱 | 更好的方式 |
|------|-----------|
| 过于宽泛 | 具体和可操作 |
| 规格不完整 | 早发现缺口 |
| 规格和实现不同步 | 更新规格或拒绝实现 |
| 没有非功能性需求 | 包括性能、安全、可扩展性 |
