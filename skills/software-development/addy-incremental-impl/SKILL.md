# Incremental Implementation (生产级增量实现)

## 概述

用薄垂直切面构建——实现、测试、验证、提交。一次一个切片。在每个切片结束时，系统是功能性的（即使只是部分功能）。使用功能标志隐藏不完整的功能。

## 核心原则

### 一个切片一个功能
```
❌ 坏：一个PR修改10个文件，添加完整功能
❌ 坏：一个PR重构一半的系统

✅ 好：一个切片 = 一个可验证的步骤
✅ 好：每个切片都使系统保持功能性
✅ 好：在合并之前测试每个切片
```

### 薄垂直切面
每个切片穿越所有层（数据库→API→UI），但只实现一小部分功能：
```
切片1：创建任务的数据库模型 + API端点 + UI表单
切片2：列表任务 + 排序
切片3：更新任务状态
切片4：搜索和过滤
```

### 功能标志
不完整的功能应该在标志后面：
```typescript
// 功能标志控制可见性
if (isFeatureEnabled('new-checkout')) {
  return <NewCheckout />;
}
return <LegacyCheckout />;
```

## 实现流程

### Step 1: 切片规划
- 将功能分解为薄的垂直切片
- 每个切片加测试需要1-2小时
- 每个切片使系统保持功能性
- 第一个切片应该建立核心基础设施

### Step 2: 实现
- TDD：先写测试
- 只实现必要的代码
- 不要提前优化
- 使用安全默认值

### Step 3: 验证
- 新测试通过
- 现有测试没有回归
- 手动验证UI（如果需要）
- 功能标志正确

### Step 4: 提交
- 原子提交（一个切片 = 一个提交）
- 清晰的提交消息
- 包括测试
- 通过CI管道

## 安全默认值
- 新功能默认禁用（功能标志关闭）
- 新的API端点需要认证
- 新的数据需要验证
- 新的UI需要加载和错误状态

## 回滚友好变更
- 数据库迁移应该是可逆的（有down方法）
- API变更应该是向后兼容的
- UI变更应该有回退机制
- 功能标志允许立即禁用

## 反合理化

| 借口 | 反驳 |
|------|------|
| "这次更改太大无法拆分" | 任何更改都可以拆分为更小的步骤。如果无法拆分，说明架构有问题 |
| "我会在下一PR中清理" | 你会在这一PR中清理。累积技术债务 |
| "这只是一个小改动" | 小改动也需要测试和审查 |
| "我会稍后添加测试" | 你不会。在这一PR中添加测试 |
