# 金鉴真人·八字排盘平台 — Harness Engineering实战案例

> 日期: 2026-06-26
> 背景: 老板发现平台不稳定，要求全方位重整

## 问题诊断

| 症状 | 根因 | Harness分类 |
|------|------|-------------|
| 引擎import错误 | `calculate_shen_qiang_ruo`函数名不存在 | 缺乏**计算型导入验证** |
| 报告一会儿好一会儿差 | 依赖大模型推理而非规则引擎 | 需要**前馈指南**固化规则 |
| 验证靠人逐行看 | 无自动验证脚本 | 需要**反馈传感器** |
| 部署手动操作 | 无CI/CD | 需要**计算型门禁** |
| API不稳定 | health端点路径不对、版本混乱 | 需要**可观测性** |

## 解决方案

### 前馈指南
- 创建 `bazi-platform-harness` 总调度，6 Phase流程
- 创建 `test_imports.py` 确保19模块可导入

### 计算型控制
- 引擎12,437行Python确定性代码
- 320条自动化测试
- 26项全量验证门禁(validate_all.py)

### 反馈传感器
- validate_all.py 自动运行（每次提交前）
- 财星§1 vs §8一致性验证

## 增量Sprint交付

| Sprint | 内容 | 验证 |
|--------|------|------|
| 1 | 引擎修复+验证强化 | ✅ 19模块/320测试/26项全PASS |
| 2 | API完善 | ✅ 8端点/health细粒度/版本5.0.0 |
| 3 | 前端21§渲染 | ✅ 515行SPA/农历/PDF/8维评分 |
| 4 | CI/CD | ✅ 三阶段/需workflow scope PAT |
| 5 | 用户系统MVP | 🔜 待启动 |

## 关键经验

1. **Prove-It模式**有效：先写复现测试→再修复→比直接猜快
2. **PAT坑**：GitHub PAT截断在记忆中会导致推不上去——存完整值
3. **验证门禁不可跳**：26项全部PASS才能提交
4. **Sprint粒度**：每个Sprint=2-3小时增量，可验证、可提交、可回滚
