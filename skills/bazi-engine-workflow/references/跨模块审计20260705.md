# 全链路跨模块审计 · 2026-07-05 方法+发现

## 审计方法论

### 第一层：模块逐行审计（铁律⑥ Step 1）
1. 加载对应skill参考 → `skill_view('bazi-xxx-analysis')`
2. 逐条对比代码逻辑 vs 原始理论（无杜撰/无遗漏/无错误）
3. 检查模块判断逻辑本身是否与九龙道长一致

### 第二层：引用链审计（铁律⑥ Step 3）
1. 确认每个模块的 import 链完整
2. 确认 `shen_label` / 喜用神等参数传递正确
3. 确认 pipeline → comprehensive → 各模块的调用顺序和参数

### 第三层：刑冲合害跨模块覆盖矩阵
**这是2026-07-05新增的方法论** — 系统性检查每种刑冲合害规则在11个事象模块中的实际使用情况。

步骤：
1. 列11个事象模块：学业/事业/财富/健康/婚姻/子女/六亲/置业/灾祸/性格/流年
2. 列12种刑冲合害规则：天干五合/天干四冲/地支六冲/三刑/六害/六破/六合/三合/半合/暗合/拱合/合化
3. 搜索每个规则关键词在每个模块中的出现 → 标注✅/❌/⚠️
4. 对❌项判断是「合理缺口」还是「实现缺失」

### 第四层：点面体系验证（老板核心方法论·必执行）
**修一个点→延伸到整个面→延伸到整个体系**
1. 修了一个Bug后，检查该Bug是否在**其他文件中有相同代码逻辑**（dual-implementation drift）
2. 检查该模块的**引用链上下游**确保改动正确传递
3. 检查该规则在**其他事象模块中是否有一致实现**
4. 选一个真实八字跑全流程验证

### 第五层：并行子Agent审计模式
当需要审计大量跨模块规则时，使用 `delegate_task` 并行派单：
- 每个子Agent专注一种规则（如：天干五合、地支六冲、三刑六害）
- 子Agent返回结构化审计报告
- Orchestrator汇总成覆盖矩阵
- 优势：互不干扰、回退时只影响单个子任务

### 第六层（🆕 2026-07-05新增）：代码路径审计
**目的**：发现「规则虽然正确，但代码实现有bug」的问题 —— 死代码、数据流错误、字符串匹配脆弱。
**触发场景**：模块审计走完前五层后，额外执行此层。尤其当模块的语义逻辑复杂（if/elif 分支多、字符串匹配多、数据在层间传递）时必须执行。

#### 检查清单

**□ ① 死代码路径检测**
逐条检查每个 if/else 分支的条件值是否**实际存在于数据源中**：
```
例：character.py L175-180
  if "叛逆型" in ss_labels    → "叛逆型" 不存在于任何十神 label 中 → 死代码 ❌
  if "开拓型" in ss_labels    → "开拓型" 不存在于任何十神 label 中 → 死代码 ❌
  if "保守型" in ss_labels    → "保守型" 不存在于任何十神 label 中 → 死代码 ❌
```
做法：列出所有合法值（如 `SHI_SHEN_CHARACTER` 中的 label 列表），逐一比对条件值。
工具：`python3 -c "种值=[v['label'] for v in DICT.values()]; print('叛逆型' in 种值)"`

**□ ② 数据流审计 — 输入参数是否被错误处理**
```python
例：character.py L149-150
  bazi_gans = [年干,月干,日干,时干]  # 含日主
  for g in bazi_gans:                  # 日主也被遍历
    ss = get_shi_shen_for_gan(g, ri_zhu)  # 日主→日主=比肩
    # ← 日主底色已被 RI_ZHU_CHARACTER 独立处理，此处重复！
```
检查要点：
- `bazi_gans` / `all_gans` 传递的是什么（含不含日主？）
- 被调函数是否期望过滤或不过滤
- 字典 key 是否全部被覆盖（无 KeyError 风险）
- `set()` 去重是否丢失重要顺序或信息

**□ ③ 字符串匹配脆弱性检测**
```python
例：character.py L182
  if "冲" in di_zhi_summary:  # 如果摘要中有"冲破""冲动"等也会误匹配
  if "三刑" in di_zhi_summary:  # 分隔符不一致可能导致漏配
```
检查要点：
- 是否可以从传字符串改为传结构化数据（dict/list）
- `in` 子串匹配是否可能误触
- 分隔符/格式是否与上游产出一致

**□ ④ 类型标签引用完整性检测**
当代码中有「标签类型」字典（如 label/type/category 字段），检查：
- 所有条件分支引用的标签值都存在于字典中
- 字典无遗漏（覆盖全部十神、全部地支、全部神煞等）
- 标签名在下游消费时的引用与上游定义一致

**□ ⑤ 动态执行验证（推荐）**
对至少一个真实输入跑模块入口函数，打印关键中间变量：
```python
# 在模块的 __main__ 测试块或独立脚本中添加
result = analyze_character(...)
print("冲突:", result["conflicts"])  # 确认非空
print("十神特质:", [t["ten_god"] for t in shi_shen_traits])  # 确认无重复
```

**六层审计后的修复验证（必做）：**
```

cd projects/bazi-platform/engine/tests && python3 -m pytest test_full_suite.py -v

# 2. 行为验证 — 确认修复生效
cd projects/bazi-platform/engine && python3 -c "
```

### 十神判定反了模式（重要！跨文件重复出现）
在**两个独立文件**中都出现了相同的十神判定反了bug：
- `shi_shen.py`（2026-07-05校准）：(`我生`,`同`)→伤官❌应为食神
- `step1_basic.py`（2026-07-05发现）：同样食伤判定反了

**根因**：写代码时记混了「同阴阳=食神/正官/正财/正印/比肩」口诀，`same_yy` 条件写反。

**防御措施**：修改任何十神相关代码后，必须同时搜索所有 `.py` 文件中 `same_yy` 相关条件判断，确保全部一致。

**检查清单（每次修改十神相关代码后执行）**：
□ 我生+同阴阳 = 食神（不是伤官）
□ 我生+异阴阳 = 伤官（不是食神）
□ 同我+同阴阳 = 比肩（不是劫财）
□ 同我+异阴阳 = 劫财（不是比肩）
□ 克我+同阴阳 = 七杀（不是正官）
□ 克我+异阴阳 = 正官（不是七杀）
□ 口诀：同食异伤、同比异劫、同杀异官、同偏异正
