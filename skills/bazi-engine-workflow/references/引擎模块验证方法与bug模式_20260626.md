# 引擎模块验证方法论与bug模式（2026-06-26实战精炼）

> **来源**：2026-06-26 session 中逐一验证每个引擎模块时发现的系统性bug模式
> **核心教训**：不验证=有bug。逐行看+逐人测=95%的bug能提前找出来。

## 一、验证方法论

### 1.1 测试策略

```yaml
【最低测试覆盖标准】

原则：必须用JSON数据源中的**所有已知八字**测试每个引擎模块，而不是只测一个。

执行方法：
  对每个引擎模块（shen_qiang_ruo.py / cai_xing.py / paipan.py / education.py等）：
  
  Step 1 — 从JSON数据源提取所有测试用例（至少5人，覆盖不同日主/身强弱/格局）
  Step 2 — 对每个模块，逐个用例验证输出与预期一致
  Step 3 — 发现一个bug → 排查是否其他用例也有相同bug（系统性bug模式）
  Step 4 — 修复后重新跑全量用例验证

测试用例选择标准：
  □ 不同性别的：男 + 女（大运方向验证）
  □ 不同日主的：金木水火土各至少1个
  □ 不同身强弱的：身强 + 身弱 + 中和 + 从弱
  □ 有燥土的（未戌）：验证燥土规则
  □ 有特殊合局的：三合/六合/刑冲
```

### 1.2 逐行代码审查法

```yaml
【不要相信「看起来对的」代码——逐行审查】

发现bug的方式不是「跑起来看结果」——更应该先逐行读代码：
  
  Step 1 — 读函数开头 -> 理解它要做什么（对照skill中的规则）
  Step 2 — 逐行扫描 -> 每个if条件/每个公式/每个循环
  Step 3 — 问自己「这行代码有没有可能错？」
  Step 4 — 发现可疑点 -> 立刻用简单测试验证

典型审查发现模式：
  「这里用了一个固定值12分——但年干是8分啊！」
  「这个循环包括了年干/月干/时干/日干——日干不应该算进去！」
  「这个hour//2直接用地支索引——但奇数小时（如21点戌时）整除2是10.5」
```

## 二、2026-06-26发现的系统性bug模式

### Bug #1: constants.py — 巳火藏干顺序错误

```yaml
【错误】
  "巳": {"藏干": ["丙","庚","戊"], ...}  
  # 庚在戊之前——但丙戊庚才是正确的（丙本气100%·戊中气60%·庚余气30%）

【影响】
  巳火藏干的十神判断全部错误——月支巳中庚金(偏财/七杀位置)被放在中气60%位，
  而戊土被放在余气30%位。直接影响：巳月的戊土/己土日主的身强弱评分。

【修复】
  "巳": {"藏干": ["丙","戊","庚"], ...}
  
【教训】
  查表数据必须与skill中的藏干标准对照（foundation-analysis有完整的藏干60/30表）
  不要相信记忆中的藏干顺序——查表确认
```

### Bug #2: paipan.py — 日柱基准日法缺少世纪修正

```yaml
【错误】
  # 直接用 2000-01-01 = 戊子日 作为基准
  # 没考虑1900-2000和2000-2100世纪的修正值差异

【影响】
  所有2000年之前出生的人日柱偏移（如1980年错误偏移1-2天）
  所有2000年之后出生的人日柱正确

【修复】
  根据不同世纪使用不同基准点：
  1900-1999: 1900-01-01 = 甲戌日
  2000-2099: 2000-01-01 = 戊子日
  
【教训】
  日干支算法有世纪修正参数（不同日历系统）
  基准日不能一劳永逸——不同世纪要不同基准
```

### Bug #3: paipan.py — 时柱计算中hour//2对奇数小时的问题

```yaml
【错误】
  shichen_idx = birth_hour // 2  
  # 戌时21点→21//2=10 ✅ 酉时17点→17//2=8 ✅
  # 但21:30呢？21.5//2=10.75→int->10 ✅ 奇数分钟是对的
  # 实际代码中用了整数hour，所以不影响

【正确验证】
  实际代码用 int(hour)//2，整数小时不可能产生小数结果
  这条其实是虚惊一场——但验证过程暴露了「测试不足」的问题
```

### Bug #4: cai_xing.py — 漏算年干+月干财星

```yaml
【错误】
  年干和月干的天干财星计分循环：只检查了「年支」「月令」「日支」「时干」「时支」
  缺少了年干和月干的独立检查！

【影响】
  如果年干或月干是财星（例如甲木日主+年干戊土偏财），这部分分数直接漏算

【修复】
  在财星计算循环中加入年干和月干的独立检查：
  - 年干（8分）：if is_cai(gan) → score += 8.0
  - 月干（12分）：if is_cai(gan) → score += 12.0
  
【教训】
  逐位置检查时最容易漏掉的就是天干（习惯性先检查地支藏干）
  检查清单必须同时包含天干和地支藏干
```

### Bug #5: cai_xing.py — 年干基础分用错

```yaml
【错误】
  对年干的财星计分用了12分（月干和时干的标准）
  但年干的基础分是8分，不是12分

【影响】
  年干财星被虚高50%（8→12分）

【修复】
  年干基础分=8分，月干=12分，时干=12分
  
【教训】
  各位置基础分必须精确：年干8·年支4·月干12·月令40·日支12·时干12·时支12
  这些是死分，背不下来就查表，不凭记忆
```

### Bug #6: shen_qiang_ruo.py — 天干比劫统一用12分

```yaml
【错误】
  天干比劫循环中，所有透干的比劫都用了12分
  但年干的比劫应该用8分不是12分

【影响】
  当年干有比劫时，身强弱被多算4分

【修复】
  分辨位置：年干比劫→8分，月干比劫→12分，时干比劫→12分
  
【教训】
  同样的错误模式：对天干比劫用了uniform分值
  不管什么位置都用同一个值 = 年干/月干/时干的分值不同！
```

### Bug #7: shen_qiang_ruo.py — 燥土规则顺序错误（壬癸盖过丙丁）

```yaml
【错误】
  # 先检查天干壬癸（湿）→再检查天干丙丁（火）
  # 但正确顺序应该是：先火后水——火引化优先
  # 且有丙丁时无论有无壬癸都当火看（火大烧不灭）

【影响】
  当八字同时有壬癸+丙丁时（如丁未月+天干壬水）：
  - 错误逻辑：壬水灭火→当土计分 ✅（实际应：丁火引化→当火不计分 ❌）
  
【修复】
  顺序改为：先检查丙丁→有丙丁则当火不计分（无论壬癸）
  无丙丁有壬癸→水灭火→当土计分

【教训】
  燥土规则的正确优先级：
  ① 天干丙/丁 → 当火（不计印分）   ← 最高优先级
  ② 无丙丁但有壬癸 → 水灭火→当土（计分）
  ③ 无丙丁无壬癸 → 默认当土（计分）
```

```yaml

   - 燥土规则顺序：条件优先级反了
```

## 五、验证标准流程

```yaml
【引擎模块验证标准流程】

第0步 — 准备测试用例
  ∎ 打开 JSON 数据源（family_bazi_data.json）
  ∎ 提取至少5个覆盖不同日主/性别的用例
  ∎ 记录每个用例的预期身强弱/财星等数据

第1步 — 逐行读代码
  ∎ 对照 skill 中的规则读每个函数
  ∎ 标注每个条件的规则来源（skill文件名+§号）
  ∎ 检查：位置倍数是正确的吗？（年干8 vs 月干12）
  ∎ 检查：循环覆盖了所有位置吗？（7个位置：年干/年支/月干/月令/日支/时干/时支）

第2步 — 单用例验证
  ∎ 以第一个人为例运行模块
  ∎ 手动计算每个位置的分数
  ∎ 对比模块输出
  ✅ 一致 → 继续第3步
  ❌ 不一致 → 排查bug

第3步 — 全量用例验证
  ∎ 对所有测试用例运行模块
  ∎ 对比 vs 数据源中预期值
  ✅ 全部一致 → 模块验证通过
  ❌ 任意不一致 → 排查bug

第4步 — 集成验证
  ∎ 将所有已验证模块串联运行（pipeline）
  ∎ 对至少3个完整八字跑全量pipeline
  ❌ 前一步正确但集成后错 → 模块间接口/数据传递bug

第5步 — 提交
  ∎ 验证通过后提交代码
  ∎ commit信息标示「验证通过：XXX/XXX/XXX测试用例全部通过」
```
