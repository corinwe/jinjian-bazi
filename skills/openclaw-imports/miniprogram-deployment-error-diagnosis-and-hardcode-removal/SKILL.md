# 小程序部署错误诊断与硬编码移除

本技能指导你系统性地诊断和修复小程序部署中的网络连接与数据加载错误，并彻底移除硬编码，确保产品交付质量。

## 何时使用此技能
*   当你部署小程序后，在真机或模拟器上遇到“加载失败”、“网络请求失败”或“没数据”等提示时。
*   当你需要将小程序代码从远程服务器（如开发环境）通过SCP拷贝到本地进行部署和测试时。
*   当你发现小程序功能（如学校详情页）因使用硬编码或模拟数据而无法显示真实API返回的数据时。
*   当你需要按照高并发、高可用的C端产品标准，对交付物进行质量审查和硬代码清理时。

## 步骤
1.  **定位并准备最新版本代码**
    *   **做什么**：在服务器上检查多个可能的小程序目录，通过文件大小、修改时间和功能完整性（如是否包含`assessment`、`international`等关键目录）来确定最完整的版本。将关键修复（如登录bug）合并到该版本中。
    *   **为什么**：确保部署基础是功能最全且已修复已知问题的最新代码，避免使用陈旧或不完整的版本。

2.  **创建并传输部署包**
    *   **做什么**：使用`tar`命令将小程序目录（可选择性包含API服务器）打包成`.tar.gz`文件。提供服务器的公网IP（例如`43.162.90.39`）和具体的SCP命令供用户拷贝到本地。
    *   **为什么**：压缩打包便于传输，提供明确的IP和命令能减少用户操作错误，加速部署流程。

3.  **诊断“加载失败”错误**
    *   **做什么**：首先检查API服务器进程是否在运行（`ps aux | grep api-server`）。然后，测试API是否能在服务器本地访问（`curl localhost:3000`），再从公网测试（`curl http://<公网IP>:3000`）。如果本地通而公网不通，问题通常是网络配置。
    *   **为什么**：此分层检查法能快速定位问题是出在服务进程、本地配置还是网络环境。

4.  **修复网络连接问题**
    *   **做什么**：执行以下三个关键修复：(1) 修改小程序全局配置（如`app.js`）中的`apiBaseUrl`，将`localhost`替换为服务器的公网IP。(2) 修改API服务器代码（如`api-server.js`），确保监听地址为`0.0.0.0`而不仅仅是`127.0.0.1`。(3) **最重要的一步**：指导用户在云服务商（如腾讯云、阿里云）的控制台中，为服务器实例的安全组添加入站规则，**开放API服务所用端口（例如3000）的TCP访问**。
    *   **为什么**：小程序在用户手机运行，无法访问服务器本地的`localhost`；API服务器需监听所有网络接口才能接受外部请求；云服务商的安全组是虚拟防火墙，默认可能阻止所有非标准端口，必须手动放行。

5.  **诊断“学校未找到”等数据错误**
    *   **做什么**：审查报错页面的代码（如`pages/school/detail.js`），查找是否存在硬编码的数据映射函数（如`getHardcodedSchoolData`）或模拟数据加载函数（如`loadMockData`）。对比硬编码的ID（如`school001`）和真实API返回的ID格式（如`qs072`）。
    *   **为什么**：这类错误通常源于开发阶段为快速实现功能而写的临时代码，在对接真实API时会产生数据不匹配，必须移除。

6.  **移除硬编码和模拟数据**
    *   **做什么**：重写数据加载逻辑，删除硬编码函数和模拟数据。改为发起网络请求（`wx.request`）调用真实API，并妥善处理成功和失败回调。删除所有相关的紧急备用文件（如`detail_emergency.js`）。
    *   **为什么**：这是交付高质量、可维护产品的核心要求。移除硬编码确保所有数据动态来自后端，使产品行为真实、可扩展。

7.  **建立质量保证与验证机制**
    *   **做什么**：创建错误记录文档（如`ERRORS.md`）记录本次迭代的问题。制定详细的验证检查清单（如`VALIDATION-CHECKLIST.md`），涵盖数据真实性、API连通性、性能、架构等维度。对修复后的代码执行清单检查，并生成验证报告。
    *   **为什么**：建立机制能避免同类错误重复发生，检查清单确保交付前所有关键质量点都经过确认，符合高标准产品要求。

8.  **打包并交付最终版本**
    *   **做什么**：将所有修复后的代码、配置文件、验证报告和部署文档打包，生成最终交付包（如`offerpath-final-validated-20260418.tar.gz`），并提供给用户。
    *   **为什么**：提供一个完整、开箱即用、且经过验证的交付物，方便用户部署和上线。

## 陷阱与解决方案
❌ **仅检查API进程存在就认为服务正常**
→ **为什么失败**：进程存在不代表服务在监听端口，或监听的地址不正确（如只监听了127.0.0.1），导致公网无法访问。
→ ✅ **正确方法**：使用`netstat -tlnp | grep :3000`命令确认服务是否在`0.0.0.0:3000`或`:::3000`上监听。同时必须从公网IP进行`curl`测试。

❌ **修改了代码但忘记更新部署包**
→ **为什么失败**：用户SCP拷贝走的仍然是包含旧bug的安装包，修复未生效。
→ ✅ **正确方法**：每次代码修复后，必须重新打包并告知用户新的文件名和SCP命令。在交付最终版前，可以删除旧的错误包避免混淆。

❌ **只修改了小程序API地址，忽略了安全组**
→ **为什么失败**：这是最常见的问题。即使小程序配置和API服务器代码都正确，云服务商层面的安全组规则会直接丢弃发往3000等非标准端口的包，导致连接超时。
→ ✅ **正确方法**：必须将**配置云服务商安全组**作为独立且关键的一步明确告知用户，并提供具体的规则参数（协议：TCP，端口：3000，来源：0.0.0.0/0）。

❌ **硬编码数据与API返回的数据结构不匹配**
→ **为什么失败**：开发时硬编码的数据Key（如`id: "school001"`）与真实API返回的Key（如`id: "qs072"`）不一致，导致前端无法正确渲染数据。
→ ✅ **正确方法**：彻底移除硬编码的数据映射逻辑。让前端页面直接使用API返回的原始字段。如果必须转换，转换规则应基于API的响应格式，而非本地假设。

## 关键代码和配置
**1. 小程序全局配置修改 (`app.js`)**:
```javascript
// 错误配置（仅本地开发可用）
// apiBaseUrl: 'http://localhost:3000/api/v1'

// 正确配置（部署到服务器）
apiBaseUrl: 'http://43.162.90.39:3000/api/v1'
// 如果通过Nginx反向代理，且代理到80端口，则可以是：
// apiBaseUrl: 'http://43.162.90.39/api/v1'
```

**2. API服务器监听地址修改 (`api-server.js`)**:
```javascript
// 错误配置（仅本地可访问）
// app.listen(PORT, () => { ... });

// 正确配置（允许公网访问）
app.listen(PORT, '0.0.0.0', () => {
  console.log(`✅ API服务器运行在: http://0.0.0.0:${PORT}`);
});
```

**3. 移除硬编码，调用真实API (`pages/school/detail.js`)**:
```javascript
// 删除类似以下的硬编码函数
// function getHardcodedSchoolData(id) { ... }

// 改为发起网络请求
wx.request({
  url: `${app.globalData.apiBaseUrl}/schools/${id}`, // 使用全局配置的API地址
  success: (res) => {
    if (res.data && res.data.school) {
      this.setData({ school: res.data.school }); // 直接使用API返回的数据结构
    } else {
      this.showError('未找到该学校信息');
    }
  },
  fail: (err) => { 
    this.showError('网络请求失败');
    console.error('API请求失败:', err);
  }
});
```

## 环境与先决条件
*   **服务器环境**：Linux服务器（如Ubuntu），具有root或sudo权限。
*   **网络**：服务器需有公网IP地址。用户需要能通过SSH访问服务器。
*   **云服务商控制台访问权限**：用户必须能登录到云服务器提供商（如腾讯云、阿里云）的控制台，以配置安全组规则。
*   **本地环境**：用户本地需安装有SCP客户端（通常与SSH一起）、微信开发者工具。
*   **Node.js**：运行API服务器需要Node.js环境（任务中版本未明确，建议使用LTS版本）。
