# read_file截断导致文件崩溃 — 2026-06-25实战记录

## 问题描述

在ALLNEW生成杨昌玉报告过程中，反复出现文件"越写越短"的现象：
1. 第一次写入：466行 ✅
2. BULK-FILL后：661行 ✅（看起来正常）
3. 再次BULK-FILL后：720行？
4. 但实际上文件内容被截断，只剩部分§

## 根因分析

`read_file(path)` 默认的 `limit` 参数为500行。当文件超过500行时：
- read_file只返回前500行的内容
- 如果将这500行内容 + 新内容 一起 `write_file`，原始文件中500行之后的内容全部丢失
- 症状：文件总行数可能看起来增加了（因为追加了新内容），但原始内容实际上丢失了

## 再现步骤

```python
# ❌ 错误做法 — 文件 >500行时丢失数据
from hermes_tools import read_file, write_file

content = read_file(path)['content']  # 只读到前500行！
sig_pos = content.find('#PIPELINE-SIG')
new_content = content[:sig_pos] + extra + content[sig_pos:]
write_file(path=path, content=new_content)  # 覆盖后只保留了前500行+新内容
```

## 正确做法

```python
# ✅ 正确做法 — 原生Python读取完整文件
with open(path) as f:
    content = f.read()  # 读取完整文件，不限长度

lines = content.split('\n')
# 定位PIPELINE-SIG
sig_idx = None
for i, line in enumerate(lines):
    if 'PIPELINE-SIG' in line:
        sig_idx = i
        break

# 在PIPELINE-SIG前插入新内容
all_lines = lines[:sig_idx] + [extra] + lines[sig_idx:]
with open(path, 'w') as f:
    f.write('\n'.join(all_lines))
```

## 验证清单（BULK-FILL后必须执行）

```bash
grep -c "#PIPELINE-SIG" {文件}  # 必须=1
grep "^## §" {文件} | wc -l    # 必须=21
grep "^## §" {文件} | sort | uniq -d  # 必须为空（无重复§）
wc -l {文件}                    # 应合理增长
```

## 经验教训

1. 任何时候对 >100行的文件做"读→改→写"操作，都要用原生Python I/O
2. read_file是一个很好的"查看"工具，但不是可靠的"编程读取"工具
3. BULK-FILL时永远先确认文件完整性和签名完整性
