## 概述

三段式全自动流水线，对标Word文档的逐字逐句处理标准：

```
音频文件 → ①永久存档 → ②逐字转写 → ③知识点提取
              ↓               ↓               ↓
       $HERMES_HOME/永维基音频库/  逐字转写.txt    知识点提取.md
         (原始音频永久保留)
```

**纯离线运行**（faster-whisper + small模型），无需API Key。

```yaml
【中断恢复流程】

① 诊断：
   - process(action='poll') → not_found → 可能磁盘满了
   - df -h / → 确认磁盘使用率
   - 查看已转写的文件数 vs 总文件数

② 腾空间（按优先级）：
   $ apt-get clean -y                    # 清理apt缓存（~100-500MB）
   $ pip cache purge                     # 清理pip缓存（~100-500MB）
   $ journalctl --vacuum-time=3d         # 清理journal日志
   $ rm -rf /tmp/whisper* /tmp/*.mp3     # 清理临时文件

③ 重启剩余文件：
   - 跳过已转写的（检查逐字转写.txt是否存在）
   - 只跑未完成的文件
   - 建议2份一组，避免中途再满

④ 验证：
   - 全部完成后检查每个目录是否有逐字转写.txt
   - 总字符数合计

口诀：进程消失别慌张，磁盘排查第一桩
     缓存清理先动手，重启续跑别跳趟
```

### 第3b步（v3.3新增）：子智能体深读审计 🔍

> **规则来源**：2026-06-24 老板要求审计「是否逐字逐句精读，提取了所有知识点」
> **执行时机**：子智能体返回知识提取结果后，skill更新前

```yaml
【审计方法】
  ① 交叉抽查：在原始转写文件中选2-3个关键段落
     - 选知识提取中提到的核心概念所在的段落
     - 对照提取结果，验证是否被准确捕捉
     - 也选一个提取中没有提及但可能重要的段落

  ② 验证阅读完整性：
     - 检查子智能体的 read_file 调用次数
     - 大型文件（>100K字）应分10-15次读取（每次500行）
     - 如调用次数明显不足 → sub-agent可能跳读了

  ③ 知识覆盖评估：
     - 提取文件章节数 vs 原始内容主题数
     - 核心方法论/口诀/案例是否全部覆盖
     - 遗漏的属于核心知识还是边角故事

  ④ 质量等级判定：
     ✅ 优秀：95%+核心知识点覆盖，逐页阅读，交叉抽查无遗漏
     ⚠️ 良好：85-95%覆盖，有小遗漏但不影响核心
     ❌ 需重做：<85%覆盖，明显跳读了关键内容

【常见审计发现】
  - 故事/案例被缩写但没有丢失核心知识点 → ✅ 可接受
  - 风水宫位细节被略过但已在其他技能覆盖 → ✅ 可接受
  - 非核心闲聊/问答环节被跳过 → ✅ 可接受
  - 核心方法论（如六合化解冲法）缺失 → ❌ 需补充
```

当有 **2份以上** 大型转写文件时，使用 `delegate_task` 并行派子智能体精读：

```python
# 策略：把所有转写文件分为最多2组并行处理
# 每个子智能体：
#   ① 读取整个文件（每次offset=1, limit=500, 逐页读完）
#   ② 逐字逐句提取知识点
#   ③ 输出结构化知识清单
#   ④ 保存到独立文件

# 关键参数：
#   - role='leaf'（不嵌套）
#   - toolsets=['terminal','file']（只需要读文件和写输出）
#   - context中必须详细说明：
#       * 转写文件的绝对路径
#       * 要提取的知识领域/关键词
#       * 输出格式要求
#       * "必须100%逐段精读，不能跳过任何内容"
#       * "使用 from hermes_tools import read_file 读取"
```

**注意事项：**
- 每个子智能体独立context，无记忆共享
- context中必须写明"respond in Chinese"，否则默认英文输出
- 子智能体的输出是self-report，读完后验证文件是否真实写入
- 大型文件（>100K字，如Part2的103K字）分~12次读取（每次500行）

### 第3c步（v3.5新增）：主Agent直接深读（替代子智能体模式）

> **适用场景**：仅1份大文件（≤150K字），且主Agent上下文窗口足够直接阅读
> **规则来源**：2026-07-04 直接读取128KB/3042行的观手看健康转写文件
> **优势**：无需delegate_task开销，所有上下文在主Agent内，分析连贯

```yaml
【主Agent直接深读工作流】

① 确认文件大小：
   wc -c -l 逐字转写.txt
   # 如果 <150KB 且 <4000行 → 主Agent可直读
   # 如果 ≥150KB 或 ≥4000行 → 改用子智能体并行模式

② 分段读取：
   read_file(path, offset=1, limit=500)     # 第1段
   read_file(path, offset=501, limit=500)   # 第2段
   ...继续直到读完
   ⚠️ 验证是否读完：最后一段的 offset+limit 应 ≥ 总行数
   ⚠️ 验证调用次数：>100K字文件应调用6-10次

③ 同步提取知识点（vs 异步子智能体）：
   - 读完一段后可在同一response中记录关键信息，但不要提前下结论
   - ⚠️ 铁律：全部读完后，再统一按知识维度重新组织提取
   - ⚠️ 铁律：不要在读完70%-80%时就输出——后文可能颠覆前文的规则

④ 审计检查（替代子智能体Checker）：
   - 验证 read_file 调用次数是否覆盖了全部行数
   - 随机选3-5个段落交叉验证提取是否准确
   - 检查是否遗漏了目录头部的元信息（置信度、模型、源文件名等）

【选择决策树】
┌─────────────────────────────────────┐
│ 文件大小？                           │
├── <150KB ─── 主Agent直读 ✅          │
├── 150KB~300KB ─── 看上下文空间       │
│  ├── 上下文宽裕 → 主Agent直读 ✅     │
│  └── 上下文紧张 → 子智能体并行 ✅    │
└── ≥300KB ─── 子智能体并行 ✅         │
└─────────────────────────────────────┘
```

## 知识提取方法论

每段音频处理逻辑与Word文档处理完全一致：

1. **逐句精读**：每段带时间戳，逐句读原文
2. **知识标注**：识别口诀/方法论/案例/特殊规则
3. ✅ **按原课程知识结构归类**（2026-06-24老板纠正）：❌ 不要自创分类名（如「经济轮廓」「物理自动化应用体系」），✅ 按老师原本的知识体系自然归类。不确定怎么分时，先按原文的话题段落分，后续再合并到现有skill体系
4. **覆盖检查**：对照现有skill体系，标记已掌握/新发现/遗漏
5. **交叉验证**：检查是否与已有体系冲突
6. **案例提取**：若有真实命例，提取案例+断语+验证

### 覆盖状态标记

| 标记 | 含义 | 后续动作 |
|:----|:------|:---------|
| ✅ 【已掌握】 | 现有技能已覆盖 | 无需操作 |
| 🔴 【新发现】 | 新知识点 | 更新对应skill |
| ⚠️ 【补充细节】 | 已有知识增加新细节 | 更新对应skill |
| 📖 【案例佐证】 | 有真实案例验证 | 可作为补充素材 |
