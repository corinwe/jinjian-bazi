#### ① 删除杜撰内容 — GONG_WX_SEX 火/金/木

**问题：** 代码中 `GONG_WX_SEX` 定义了五行全量映射：
```python
GONG_WX_SEX = {"土": "配偶肾功能偏弱（土克水）", "水": "肾功能强",
               "火": "精力旺盛", "金": "适度", "木": "和谐"}
```

九龙道长原始理论只覆盖了土和水（土克水→肾功能偏弱、水旺→肾功能强），火/金/木的性功能断语**无原始依据**。

**修正：** 删除火/金/木断语，保留土和水。

**⚠️ 2026-07-05审计发现「修正过正」陷阱：** 校准说「保留土和水,删除火/金/木」→ 代码实现中 GONG_WX_SEX 被整个删除,土/水的性功能断语也未实现。每次按校准修正代码后必须用真实案例验证「保留A+B,删除C」没有变成「删除A+B+C」。

**强制验证清单(每次校准修正后):**
```yaml
□ 校准修正范围是否比skill规定的更宽？
   例：skill说「删除火/金/木,保留土/水」
    → 代码只删了火/金/木？还是删了整个dict？
□ 被保留的部分(土/水)是否确实在代码中存在？
□ 配偶宫五行=土时是否有「肾功能偏弱」断语？
□ 配偶宫五行=水时是否有「肾功能强」断语？
```

#### ② signal_detail 字段 key 名不一致 Bug

**问题：** `analyze_marriage()` 的 `signal_detail` 读取 `w['reason']`（单数），但 `_calculate_marriage_windows()` 输出的字典 key 是 `"reasons"`（复数）。结果 `signal_detail` **永远为空**。

**修正：** `w['reason']` → `w['reasons']`；且 `reasons` 是 list，取 `reasons[0][:15]` 截取第一个理由。

**教训：** 修Bug后必须查所有引用方 + 确认调用机制（铁律⑨）。
- `signal_detail` 被 `_gen_detail_analysis.py` 和 `pipeline_v5.py` 读取
- `marriage_v2.py` 只被 `pipeline_v5.py` import，但 `_gen_detail_analysis.py` 间接依赖

### 深层机制

```yaml
入赘的本质：男方身弱不胜财 → 财（妻星/妻家）反克自身
  不是简单的"怕老婆"，而是：
  - 财星能量太强 → 妻家提供的资源/条件远超男方自身能力
  - 身弱接不住 → 婚姻关系中处于被支配地位
  - 印星被克 → 连原生家庭也帮不上忙（财克印）
  - 最终结果：男方"入"到女方家，以换取妻家资源的支撑

所以入赘不是道德问题，是能量匹配问题：
  身强财旺 → 男方能驾驭 → 妻家资源为己所用
  身弱财旺 → 男方被财反克 → 被妻家所控
```
