```yaml
模式① — 子代理发明「排除规则」：
  现象：子代理写"巳中庚不算用""藏干财星不算入用"
  原因：context中体用定义不完整，子代理自行「优化」
  修复：见 §3A.9铁律①「藏干财官一律算用，禁止排除」

模式② — 口诀引用不校验条件：
  现象：四柱皆带财(年柱无财)、身弱不担财(实际身强)
  原因：直接引用口诀不动脑
  修复：见 §3A.9铁律③ + SOP口诀条件校验步骤
  口诀：口诀引用先校验，条件不符不准用；四柱皆财查年柱，身弱担财看强弱

模式③ — 算术错误：
  现象：12+7.2+12+24=40.8（应为55.2）
  原因：手算不校验
  修复：见 Phase 5.1 引擎数据对齐检查
  口诀：算术写完再检查，四项加和看对错；引擎数据做基准，报告数字要匹配

模式④ — 大运真假误判：
  现象：戊子无戊(实藏申余气)、甲申无甲(实藏亥中气)

  修复：见大运真假判定精确规则（藏干余气也算有）

### 🚨 前端开发致命陷阱 — JS语法错误 + 下拉框失效（2026-06-26实战沉淀）

> **致命教训**：本场对话中，我将日期输入从文本框改为下拉框(`<select>`)，同时在JS中添加了 `initDateSelects()` 函数。但完全没生效——用户说「点框没反应」。
> **根因**：`renderReport()` 函数中有一行模板文字语法错误未闭合——`\`${year年${month}月${day}日\`` 缺少3个右花括号`}`。因为所有JS代码在同一个`<script>`块中，**一个语法错误导致整个脚本块不执行**，包括已经写对的 `initDateSelects()`。
> **教训**：JavaScript在`<script>`块中是整体编译的——任一处的语法错误会杀死所有代码。不是"局部不影响"。

```yaml
【⚠️ 模板文字嵌套陷阱 — JS特有风险】

问题模式：
  `\`${year}年${month}月${day}日\``  ✅ 正确，三个独立的模板表达式
  `\`${year年${month}月${day}日\``    ❌ 错误，试图把"年""月""日"塞进变量名
  
  错误写法看起来"合理"（中文也能当变量名的一部分），但实际上：
  `${year年` → 试图引用变量 `year年`（不存在）
  `${month}` → 正确的模板表达式
  `${day}`   → 正确的模板表达式
  缺少一个 `}` 来闭合最外层的 `${year年...}` → 语法错误

防御规则（写在代码里，不靠记忆）：
  ✅ 每个 `${...}` 独立出现：`${year}年${month}月${day}日`
  ❌ 禁止 `${var1}text${var2}` 改成 `${var1textvar2...}` 的形式
  ✅ 写完模板文字后，逐个数 `{` 和 `}` 数量是否一致
  
  🚨 通用的JS语法检测方法（不需要打开浏览器控制台）：
  ```bash
  # 检查文件中的大括号数量
  grep -o '{' index.html | wc -l
  grep -o '}' index.html | wc -l
  # 不相等 → 必有语法错误
  
  # 提取<script>内容单独验证
  curl -s http://localhost/ | python3 -c "
  import sys,re
  html = sys.stdin.read()
  m = re.search(r'<script>(.*?)</script>', html, re.DOTALL)
  js = m.group(1)
  print('{ =', js.count('{'), '} =', js.count('}'))
  print('✅' if js.count('{') == js.count('}') else '❌ 不匹配'
  "
  ```

【⚠️ select下拉框"点不动"的诊断思路】

当用户说"点下拉框没反应"时，按顺序排查：

  Step 1 — 服务器是否在返回新页面？
    curl -s http://localhost:8000/ | grep -c 'select id="year"'
    → 0 = 页面还是旧版（缓存问题）
    → 1+ = 页面已更新，JS有问题

  Step 2 — JS是否有语法错误？
    grep -o '{' index.html | wc -l VS grep -o '}' index.html | wc -l
    → 不相等 → 某处括号不匹配
  
  Step 3 — 用逐函数括号检查精确定位（本场对话实战方法）：
    ```python
    # 提取所有function，逐个检查括号数
    for fn_name in ['initDateSelects','doAnalyze','renderReport',...]:
        idx = js.find('function ' + fn_name)
        body = js[idx: 下一个function或文件尾]
        o = body.count('{')
        c = body.count('}')
        print(f'{"OK" if o==c else "MISMATCH"} {fn_name}: {o} vs {c}')
    ```
  
  Step 4 — 定位到具体行后，找模板文字中的 `${}` 嵌套
    → 最常见的JS括号不匹配来源 = 模板文字 `${...}` 写错

口诀：JS语法错一个，整个脚本全死掉；下拉框没反应先查括号配不配；
      模板文字最易错，${}逐个独立写，别逞能玩嵌套
```

### 为什么必须用三引擎架构

| 旧做法（LLM自觉） | 问题 | 新做法（代码硬约束） |
|:---|:---|:---|

| manual步骤写在SKILL.md里 | 阅读耗时长，优先级不够 | 脚本化验证门禁，不可绕过 |

### 🚨 0617致命教训：LLM禁止做任何数学计算（工程化一致性的根本保障）

> **致命错误复盘**：本会话中，6个人的财星分数通过delegate_task子agent各自计算，结果出现了6种不同的计算方式（有人用了40分×%、有人用了12分×%、有人把劫财算进财星、有人漏算了年支财星）。每次出错的根源都是：**让LLM做了本应由代码完成的确定性计算**。

```yaml

  Step 3 — 生成一份「核心数据总表」（master data table）作为唯一数据源
  Step 4 — 每个delegate_task子agent从总表获取数据（不是自己算！）
  Step 5 — 验证脚本检查：报告中所有数字是否与引擎JSON一致

❌ 错误做法：子agent各自做计算 → 6个人6个算法 → 全错。

### 🚨 ALLNEW全量重写时：防止分批次写入文件导致§头部丢失（2026-06-25新增·本会话经验）

> **致命错误**：本会话全量重写老板v26.0报告时，第一次写入§1~§6（约660行·21KB），第二次写入§7~§21（约700行·40KB）。因使用 `write_file`（覆盖式写入），第二次写入完全覆盖了第一次的内容，导致§1~§6全部丢失。最后需要合并为单次写入。

```yaml
【强制规则】当报告内容需分多次写入时：

Step 0 — 单次写完方案（推荐）
  ✅ 如可能，在单个 `write_file` 调用中完成全部内容
  ✅ 容量允许时，一次性写入完整报告（40-50KB以内可一次完成）

Step 1 — 如必须分多次（容量超限导致）→ 采用**追加模式**而非覆盖
  ❌ write_file → 覆盖式写入 → 否则每次会清空之前的内容
  ✅ terminal('cat >> {path}') → 追加写入新内容到文件末尾
  ✅ 或用 read_file 先确认旧内容是否还在，再用 write_file 一次性写入全量

Step 2 — 验证（强制）
  □ 文件头部含 # 标题和§1内容（没有被覆盖掉）
  □ 文件尾部含§21和签名行
  □ 全文grep关键词确认中间§全部存在

口诀：全量重写大报告，一次性写完最可靠
      分段追加用cat，write_file会覆盖掉
      写完先看头到位，尾部签名不能少
```

### 子agent context铁律

每个子agent的context必须包含（缺一不可）：
1. ✅ 基本八字信息（四柱/日主/身强弱/喜忌/忌神）
2. ✅ 引擎JSON路径（/tmp/bazi_batch/{name}.json）
3. ✅ **🚨 天干组成验证（0617致命错误防御 — 必须从引擎JSON提取，参见 bazi-foundation-analysis §天干描述精度规则）**
   - 不可在context中写「三壬一癸」「三壬透干」等概括性描述
   - 必须从引擎JSON `data['十神']` 字段提取具体天干：
     ```python
     tiangan = {
       '年干': data['四柱']['年柱'][0],
       '月干': data['四柱']['月柱'][0],
       '时干': data['四柱']['时柱'][0],
     }
     # 统计各天干出现次数
     壬_count = sum(1 for g in tiangan.values() if g == '壬')
     癸_count = sum(1 for g in tiangan.values() if g == '癸')
     print(f"年{tiangan['年干']} 月{tiangan['月干']} 时{tiangan['时干']} = {壬_count}壬{癸_count}癸")
     ```
   - context中写出精确天干组成（如「年壬+月癸+时壬 = 两壬一癸（含日主共两壬两癸）」）
   - 0617致命教训：琼戌时实际天干为壬癸癸壬（两壬两癸），但context写「三壬一癸」→ 报告全文13处出错 → 用户指出修正
   - 口诀：天干组成别凭忆，引擎JSON逐字查；年干月干时干数，壬几次癸几回记清楚
4. ✅ **🚨 大运年份+起运年龄（必须从引擎JSON直接提取，禁止自行计算！2026-06-24新增·4份全错教训）**\n   - 子agent**不得自行计算任何大运年份**——即使给出了起运年龄，子agent的Q4进位公式与引擎不一致也会算错\n   - 正确做法：从引擎JSON提取 `data['大运']['序列']`，**直接使用**每步的起始年份和终止年份\n   - context中必须写明大运序列数据（填入引擎实际数据）：\n     ```\n     ⚠️ 大运年份从引擎JSON提取，禁止自行计算！\n     | 大运 | 年份范围 |\n     | :---- | :-------- |\n     | 甲申 | 1981~1990 |\n     | 乙酉 | 1991~2000 |\n     ...\n     ```\n   - 示例格式：`起运年龄 = 0.38（0岁4个月17天），实用起运年份=1981年`

5. ✅ **🚨 婚姻分析必须注明命主性别（0617D级致命错误防御）**

## 🚨 已知缺陷·验证器误报

`bazi-report-validator.py` 的检查有3个已知误报模式

### 误报1：事件时序 — 未来年份被误判
- **现象：** 事件总表中的未来年份（如2040年「2040→发财」）被误判为"出生年份晚于2030年"
- **根因：** 检查逻辑过于宽松，只搜索年份附近是否出现"出生"二字
- **临时绕过：** 忽略"时序矛盾"中明确指出"出生年份(20XX)晚于2030年"的条目（这些是事件年份不是出生年份）
- **真实时序矛盾应关注：** 离婚<结婚、子女<父母、离婚与结婚同年等

### 误报2：事件类型检测 — 表情图标匹配失败
- **现象：** 验证器报告 `❌ X类事件: 0次`（如学业/事业/结婚等9类全部0次）
- **根因：** 验证器用正则匹配表情图标（📚💼🤑等），但子agent生成的报告使用 **文字标签**（如 `**学业**` `**婚姻**` ）而非图标
- **绕过：** 手动抽查事件表确认全部9类已覆盖即可，验证器的图标匹配是已知false alarm
- **注意：** 子agent生成报告时若使用图标，反而可能和其他板块的格式冲突

### 误报3：「同年结婚和离婚」—— 预警类文本触发
- **现象：** 验证器报告 `❌ 同年结婚和离婚: {2020}` 等
- **根因：** 验证器搜索年份附近同时出现"结婚"和"离婚"四字，但白话预警分析中可能写「……严重时可能分居或离婚」——这是⚠️预警而非实际事件
- **绕过：** 确认报告中结婚年份是实际事件、离婚仅出现在未来预警中，忽略该警告
- **正确关注：** 只有当事件表中明确在同一行或相邻行出现「结婚」和「离婚」时才是真实矛盾
