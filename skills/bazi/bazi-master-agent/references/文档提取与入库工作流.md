
### 1.2 PDF文件（扫描件/图片PDF）

使用 `ocr-and-documents` skill 中的 marker-pdf 或 pymupdf 提取：

```bash
# 如果fitz提取为空或乱码，说明是扫描件

```

### 1.3 docx文件（Word格式）

使用 zipfile + xml.etree.ElementTree：

```python
import zipfile, xml.etree.ElementTree as ET

with zipfile.ZipFile(path) as z:
    xml_content = z.read('word/document.xml')
    root = ET.fromstring(xml_content)
    ns = {'w': 'http://schemas.openxmlformats.org/wordprocessingml/2006/main'}
    texts = [t.text for t in root.iter('{http://schemas.openxmlformats.org/wordprocessingml/2006/main}t') if t.text]
    text = '\n'.join(texts)
```

注意：`read_file` 无法读取二进制 docx，必须用此方式提取。

### 1.4 批量提取Python脚本模板（多文件并行）

```python
import zipfile, xml.etree.ElementTree as ET, os

def extract_docx_text(path):
    with zipfile.ZipFile(path) as z:
        xml_content = z.read('word/document.xml')
        root = ET.fromstring(xml_content)
        texts = []
        for t in root.iter('{http://schemas.openxmlformats.org/wordprocessingml/2006/main}t'):
            if t.text:
                texts.append(t.text)
        return '\n'.join(texts)

# 批量处理
new_docs = {
    'NN-主题_YYYY-MM-DD.txt': 'doc_xxx.docx',
}
for out_name, docx_name in new_docs.items():
    src_path = os.path.join(base, docx_name)
    if os.path.exists(src_path):
        text = extract_docx_text(src_path)
        with open(os.path.join(kb_raw, out_name), 'w', encoding='utf-8') as f:
            f.write(text)
```

## 二、KB审计流程

### 2.1 匹配规则

1. 从文档标题提取前3个核心主题词（如"十神""课程""分析"）
2. 遍历 KB `00-原始素材/` 目录下的所有文件名
3. 若文件名包含全部核心词 → ✅ 已存在
4. 模糊匹配时，读取KB文件前500字符确认

**⚠️ 陷阱：主题词匹配容易误判（false positive）**
```yaml
问题：多份文档的标题共享相同高频词（如"八字""课程""分析"），
      导致所有文档被匹配到同一份KB文件。
      
正确做法（三段确认法）：
  第一段：提取文档标题中的时间关键词（如"2026-06-01"）+ 主题词（如"九字真言"）
  第二段：按文件名逐一精确匹配，而非关键词模糊匹配
  第三段：对匹配结果有疑问时，读取KB文件开头确认实际内容
  
  示例（错例回顾）：
    doc标题: 九字真言及八字命理分析_2026-06-01
    匹配到: 文字记录_八字能量加强及灾祸分析_2026年6月1日.txt  ← ❌ 仅因"八字"+"06-01"匹配
    正确: 该文件不在KB中 → 应标记为❌未入库
```

💡 **更可靠的审计方法**：用 `ls -la --sort=time` 列出所有文件的时间信息，  
按本次会话中新上传的文件（时间戳最新）精确圈定需要处理的范围，  
然后逐一确认每个新文件 vs KB 文件的状态。

### 2.2 原始素材编号

```
命名规则：NN-主题_YYYY-MM-DD.txt
编号说明：
  01-08：早期PDF批次（Day1/Day2 + 六月第一周课程）
  09-18：docx批次（九字真言/流年/财运/十神/四柱/学历等）
  19起：后续新增素材，保持连续性
```

## 三、入库后处理

### 🚨 3.0B 【致命遗漏防御·2026-06-22新增】全量交叉验证门禁（提取后强制审计）

> **致命错误复盘**（2026-06-22）：当晚处理5份音频文件（桃花过三关答疑/八字配婚/21种能量补充法上下/过三关实战绝招），提取全部知识点保存为独立文件后，**仅将推测与现有技能明显匹配的3份（八字配婚→marriage、过三关→fortune）整合到了技能中**。剩下2份（21种能量补充法上+下，共252分钟/208K字/38个知识点）被遗漏——知识点文件虽然存在，但**从未融入任何SKILL.md**。直到老板第二天质问"全部审计过没有"才发现47%的知识点仍是孤立文件。
>
> **根因**：缺少"提取后所有知识点文件必须找到技能归宿"的强制验证环节。
>
> **铁律**：**知识点提取 ≠ 技能整合。保存到知识库 = 半成品。融入SKILL.md = 完成品。**

```yaml
【强制流程】每次从N个源文件提取知识点后，必须执行以下审计：

Step 0 — 列出所有本次生成的提取文件
  → find 知识库/ -name "*知识点提取*.md" -newer {本次开始时间}
  → 得到一个文件列表：F = [f1, f2, ..., fn]

Step 1 — 对每个f∈F，问3个问题找技能归宿：
  □ 问题1：f中的核心知识点已存在于哪个已加载skill中？
    例：21种能量补充法的核心是「化解/补能」→ bazi-remission-methods
    例：八字配婚课程的核心是「合婚」→ bazi-marriage-analysis
  □ 问题2：如果已有skill覆盖了大部分内容但缺细节 → patch该skill的对应章节
  □ 问题3：如果没有任何skill覆盖 → 创建新skill或追加到最接近的skill

Step 2 — 逐一验证整合完成（不可跳过！）
  □ 对每个f∈F，用search_files验证其核心关键词已出现在至少1个skill中
    例：search_files(pattern="珠宝首饰.*性价比|金发晶|阿富汗玉", 
                     path="skills/bazi-remission-methods/SKILL.md")
    ✅ 有命中 → 已验证，该文件的知识点已整合
    ❌ 无命中 → 🚨 阻断！该文件的知识点尚未整合进任何skill！
      处理：立即整合，不得判为"已存知识库就行"

Step 3 — 整合率达标门禁
  □ 整合率 = (已整合文件数 / 总提取文件数) × 100%
  □ ≥ 80% → ✅ 通过，允许推库
  □ < 80% → ❌ 未通过！找出未整合的文件逐一补上再推库

Step 4 — 全部通过后，再执行推库

⚠️ 常见迷思（必须纠正）：
  ❌ "知识点提取文件已经保存到知识库了，知识已经在了" 
     → 这是半成品，不是完成品。保存 ≠ 整合
  ✅ "知识点提取文件已经保存，且每个知识点都已出现在SKILL.md中"
     → 这才是完成品

口诀：
  提取不是终点站，整合技能才算完
  每份提取文件走一遍，核心概念搜skills见
  整合率过八成才能推，半成品知识库不能留
```

### 🚨 3.1A 自动应用体系集成（致命遗漏防御·2026-06-11新增）

> **踩坑记录（2026-06-11）**：处理第一份生死劫文档时，只做了原始素材入库+skill知识更新，遗漏了「将新知识对应的自动检查逻辑写入 bazi-auto-verify」。被用户追问"自动应用体系建立了吗"才补上。
> **教训**：知识入库 ≠ 建立自动应用。必须显式检查新知识是否需要集成到自动验证流程中。

**每次知识提取后的强制检查：**

```yaml
□ 检查① — 本次提取的知识是否有「流程型」内容（检查步骤/判断流程/验证规则）？
  → 是 → 写入 bazi-auto-verify 的对应 Step 中
  → 否 → 跳过

□ 检查② — 本次提取的知识是否有「门禁逻辑」（如果不做X则出错）？
  → 是 → 写入 bazi-auto-verify 的门禁系统中（三道物理门禁/Step 6.5等）
  → 否 → 跳过

□ 检查③ — 本次提取的知识是否需要作为「输出前检查清单」的一项？
  → 是 → 追加到 bazi-auto-verify 的「输出前强制检查清单」
  → 否 → 跳过

□ 检查④ — 三步都无 → 确认本次不需要自动应用集成

口诀：
  知识入库还不够，自动应用也得走
  流程门禁检查单，三步验证不遗漏
  用户问起才补上，那就已经丢脸了
```
