# C端八字产品UI/UX铁律（2026-06-26深夜更新）

> 根据老板多次批评「傻执行」沉淀为永久铁律。
> 每个UI/UX决策前，先读这个文件。

## 一、信息架构铁律

### 1.1 根路径设计
- 根路由 `/` 必须直接返回首页HTML（`FileResponse` / `index.html`）
- 禁止用 `/static/index.html` 这种路径 — 用户只应该记 `IP:PORT`
- 如果必须挂载静态文件，用 `app.mount("/static", ...)` + 根路由 `FileResponse`

### 1.2 首页设计（最重要的原则）
- **首页第一屏 = 输入框**，用户进来就能填，不用点任何按钮
- ❌ 不要放功能卡片（自动排盘/21§报告/30年流年/PDF下载）— 用户不想先看广告
- ❌ 不要"开始排盘"按钮作为中间跳转 — 每多一步=流失率翻倍
- ✅ 用户打开页面，看到的直接是：姓名 + 出生日期 + 性别 + 时辰

### 1.3 报告展示（2026-06-26新增）
- ❌ **永远不要使用Tab导航** — 在手机上Tab难以操作，部分Tab点了没反应
- ✅ 使用**纵向滚动卡片**，从上到下展示：总览→格局→身强弱→用神→大运→财富→事业→学业→婚姻→子女→健康→三决断→五行建议
- ✅ 先出总览卡片（八字/身强弱/格局/用神/财星/最佳大运 + 综合判断条），像体检报告摘要
- ✅ 有数据才展示该卡片，无数据自动跳过
- ❌ 底部不要显示引擎版本号（"金鉴真人·全量报告引擎 v4.0"）— C端用户不关心

### 1.4 日期输入
- ✅ 用一个 `<input type="date">` 代替三个分开的选择框或文本输入框
- ✅ `color-scheme:dark` 适配暗色主题
- ❌ 不要用三个分开的 `<select>` 下拉框（年/月/日各一个）— 浪费时间
- ❌ 不要用 `<input type="number">` 文本框 — 手机端容易输错
- ✅ 范围限定：`min="1900-01-01"`，不用前端校验（date picker原生限制）

### 1.5 农历支持
- ✅ 默认支持阳历/农历切换
- ✅ 农历模式显示提示：「日期选择器中选对应的农历日期」
- ✅ 后端自动转换：`lunar_to_solar()` 后再排盘

### 1.6 时辰输入
- ✅ 用 `<select>` 下拉框，12个时辰 + 「不清楚时辰」选项
- ✅ 不清楚时辰 → 默认子时(23:00)，显示提示「将按子时估算」

## 二、前端开发铁律

### 2.1 事件绑定（2026-06-26新增）
- ❌ **不要用 `onclick="funcName(param)"` 属性** — 移动端兼容性不稳定
- ✅ 用**事件委托**或 `element.onclick = function` / `addEventListener`
- ✅ Tab/按钮用 `data-*` 属性传参，`e.target.closest()` 定位元素
- ✅ 给所有可点击元素加 `cursor:pointer`

### 2.2 数据渲染（2026-06-26新增）
- ❌ **永远不要直接 `${value}` 渲染对象** — 会产生 `[object Object]`
- ❌ **永远不要 `array.join(',')` 渲染对象数组** — 同样产生 `[object Object]`
- ✅ 检查 `typeof value === 'object'`，递归渲染嵌套对象
- ✅ 对象数组 → 格式化为表格（`<table><tr><th>...</th></tr>...</table>`）
- ✅ 标量数组 → `join(', ')` 展示

### 2.3 JSON泄露防护（2026-06-26新增）
- ❌ 禁止在UI中出现任何原始JSON / 代码片段 / 数据结构
- ✅ 每个API字段必须有对应的格式化渲染函数
- ✅ 万能兜底方案：`renderKV(data)` 递归格式化，不暴露原始JSON
- ✅ 定期检查API返回数据中是否有对象字段被直接当作字符串渲染

### 2.4 打印/PDF
- ✅ 用 `window.print()` 配合 `@media print` CSS 样式，支持"另存为PDF"
- ✅ 打印样式：白底黑字，所有section展开，page-break-inside:avoid

## 三、后端开发铁律

### 3.1 排盘引擎（2026-06-26新增）
- ✅ **永远使用 `bazi-engine.py`**（ephem天文库+节气精确计算）进行排盘
- ❌ 禁止自己写 `paipan.py` 替代已验证引擎 — 月柱/日柱计算极易出错
- ✅ 通过 `subprocess.run` 调用 `bazi-engine.py --json` 获取JSON输出
- ✅ 时辰索引映射：`hour // 2` → shichen_idx（0子1丑2寅3卯...）

### 3.2 评分引擎
- ✅ `shen_qiang_ruo.py` 和 `cai_xing.py` 按九龙道长原始规则实现
- ✅ 规则逻辑：逐个位置检查，满足条件加分，不满足不加分
- ✅ 本气100% / 中气60% / 余气30% 逐步计分
- ❌ 禁止自己推导评分公式 — 用现有的已验证引擎

### 3.3 pyc缓存坑（2026-06-26新增）
- 修改 `engine/*.py` 后必须执行：`rm -rf __pycache__/`
- 且必须**重启服务器**（`pkill -f main.py` + 重新启动）
- 否则旧pyc缓存会导致改了代码但没生效，以为修好了实际还是旧的

## 四、调试优先级（2026-06-26新增·老板多次强调）

当用户报告「数据不对」时，调试顺序是：

```
1. 直接跑引擎 → python3 engine/xxx.py     ← 先确认引擎本身对
2. 直接调API  → curl .../api/v1/analyze    ← 再确认API返回对
3. 查前端渲染 → 浏览器看页面展示           ← 最后检查UI
```

**永远不要**在数据是错的情况下重写UI。先修引擎，再修API，最后调界面。
