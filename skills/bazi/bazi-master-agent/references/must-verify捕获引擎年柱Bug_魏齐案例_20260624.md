# 物理机制实战验证案例：must-verify捕获引擎年柱Bug

> **发生时间：** 2026年06月24日
> **人物：** 魏齐（男，公历2020年1月28日 20:00 戌时）
> **涉及脚本：** `bazi-pipeline.sh` → `bazi-must-verify.sh`

---

## 事件经过

按全流程物理保证体系运行 `bazi-pipeline.sh` 处理魏齐八字时，must-verify发现：

| | 引擎输出 | 官网输出 | 正确值 |
|:--|:--------|:--------|:------|
| 年柱 | 庚子 ❌ | 己亥 ✅ | 己亥 |
| 月柱 | 己丑 ❌ | 丁丑 ✅ | 丁丑 |
| 日柱 | 庚午 ✅ | 庚午 ✅ | 庚午 |
| 时柱 | 丙戌 ✅ | 丙戌 ✅ | 丙戌 |

年柱差了一年（庚子→己亥），月柱也差了一个月（己丑→丁丑）。

## 根因

2020年1月28日**还不到立春（2月4日）**。在八字体系中，年柱以立春为界——立春前仍是上一年的年柱（己亥年）。但引擎的节气计算在此处有bug，提前切换到了庚子年。

月柱同理：1月28日还在大寒（1月20日）到立春（2月4日）之间，应为丑月（丁丑月），但引擎错误地算成了寅月（己丑月）。

## 意义

这是**物理机制（must-verify.sh）成功拦截数据错误**的典型案例：

1. ✅ pipeline作为唯一入口——跑了must-verify
2. ✅ 引擎与官网对比——立即发现不一致
3. ✅ 自动报错并停止——阻止了基于错误八字的后续分析
4. ✅ 正确数据被记录——官网八字（己亥 丁丑 庚午 丙戌）成为最终版本

如果没有must-verify的引擎⇔官网对比功能：
- 我会使用引擎的庚子年 ❌
- 所有后续分析（身强弱/大运/十神/性格）都基于错误的年柱和月柱
- 整份报告作废

## 教训

**引擎不是万能的。** 年柱立春切割、月柱节气切割是八字排盘中最容易出错的环节。must-verify的「发动机+官网双通道」对比设计不是冗余，是必要的安全检查。
