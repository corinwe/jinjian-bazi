# 金鉴真人·外部知识萃取方法论

## §1. 萃取流水线总纲

### 🚨 铁律零：先研究工具方案，再动手执行（2026-07-10 老板校准）

> **教训**：老板发来21个抖音视频要求萃取，我试了几个方法失败就开始做"主题级缺口分析"。老板指出：**你要先研究工具方案，不要空转。**
>
> **铁律**：接到"提取某平台内容"任务时，第一步是调研可用方法栈（至少5种），按优先级逐一尝试。不要用一种方法试两次以上——不行就换方法。

```yaml
调研方法栈:
  ① 列出所有可能的提取路径（至少覆盖5种不同方法）
  ② 快速评估每条路径可行性
  ③ 选最佳路径执行
  ④ 失败 → 记录原因 → 换下一条
  ⑤ 所有路径失败 → 接受技术限制 → 转主题分析法
```

### 1.1 五步萃取法

```
Step 1 — 获取源内容
  ├─ 抖音视频: Playwright浏览器爬取标题/描述/合集信息
  ├─ 公众号文章: 直接复制文案或API抓取
  ├─ 课程笔记: 原始文档提炼
  └─ 无法直接获取: 从标题/合集目录推断主题，直接做主题级缺口分析

Step 2 — 全量对比（vs 金鉴真人体系）
  ├─ 提取源内容的关键知识点列表
  └─ 逐条对比以下技能库:
      · bazi-foundation-analysis（基础规则 4257行）
      · bazi-wealth-analysis（财富体系 2591行）
      · bazi-fortune-analysis（格局喜用 2959行）
      · bazi-marriage-analysis（婚姻 1203行）
      · bazi-liunian-analysis（流年 1507行）
      · 其他事象技能

Step 3 — 分级评估
  覆盖率分级:
    ✅ 已覆盖 — 不处理
    🟡 部分覆盖 — 补充扩展条款
    🔴 缺失 — 新增独立章节或独立技能

Step 4 — 缺口整合
  P0: 核心理论缺口 → 立即新增到对应技能
  P1: 重要应用缺口 → 排入补充队列
  P2: 小众/特殊话题 → 标注待补充

Step 5 — 双库同步
  ├─ 知识库 → 归档原始萃取笔记 (.md)
  └─ 技能库 → 将新增规则整合到对应SKILL.md
```

### 1.2 萃取的三个原则

| 原则 | 内容 | 反例 |
|:-----|:------|:------|
| **原则A** | 不重复录入已有规则 | 视频说"食伤生财"→已覆盖，不录入 |
| **原则B** | 只录有增量价值的内容 | 同一概念但新命名/新框架→录入命名/框架 |
| **原则C** | 交叉验证后再融合 | 不能直接照搬，必须与九龙道长原始理论比对 |
| **🆕原则D** | **只提一次也必录** | 即使该概念只在文档中出现一次，只要是通用规则/原理，必须提取。不能因为"只提一次"就遗漏。「哪怕他只提到一次但是只要是通用的规则和原理都不能放过」— 老板铁律 |

## §2. 抖音视频提取技术（方法栈 · 2026-07-10验证）

### 2.0 完整方法栈（按优先级排列）

| 优先级 | 方法 | 状态 | 说明 |
|:------|:-----|:-----|:------|
| ① | Playwright浏览器自动化 | ⚠️ 可行 | 可获标题+描述+合集目录，点"展开"按钮获取更多内容。用手机设备模拟(iPhone UA)效果更好 |
| ② | **第三方转录工具** (anxinxia等) | ⚠️ 需非黑名单IP | `jiexi.anxinxia.com/tiqu` 提供在线视频文案提取(ASR语音转文字)。工作流：解析视频→下载音频→转文字。该工具本身使用faster-whisper/类似ASR引擎，需从非黑名单IP调用(服务器IP被抖音屏蔽时也会失败) |
| ③ | yt-dlp + 登录态cookies | ⚠️ 需cookies | 有登录态cookies后可下载音频→Whisper转录得到完整口播文案。faster-whisper已安装可用 |
| ④ | 第三方解析API | ❌ 接口过期 | `douyin` python包(2022) amemv.com接口已失效 |
| ⑤ | 抖音REST API / amemv.com内部API | ❌ 需签名参数 | 需要X-Bogus签名+app版本参数，黑名单IP即使有签名也被拦截 |

**核心结论**: 
- 获取完整口播文案的完整流水线: `获取视频URL → 解析下载地址 → ffmpeg提取音频 → faster-whisper转文字`
- 关键瓶颈: 服务器IP若被抖音列入黑名单，解析下载地址这一步在所有方法上都会失败
- 解决方案A: 用非黑名单的服务器做中转
- 解决方案B: 让老板直接从手机端使用anxinxia类工具提取粘贴
- 无cookies/非黑名单IP时，只能用Playwright获取标题/描述/合集，做主题级缺口分析

### 2.1 Playwright方案（当前最可行）

### 2.1 适用场景

抖音视频文案提取，获取标题/描述/合集目录/作者信息。

### 2.2 使用方式

```bash
# 直接运行提取单条视频
python3 scripts/douyin-extractor.py <video_id>

# video_id从短链解析: v.douyin.com/xxx → 重定向 → video/xxx
```

### 2.3 已知限制

- Douyin反爬严格，Playwright可提取页面标题+描述+合集目录，但无法获取视频内口播文案
- 文案获取需：下载音频→Whisper语音识别（技术可行但耗时较长）
- 合集目录本身已是高价值信息——从标题可知视频主题范围

### 2.4 优先替代方案

当技术手段无法获取完整文案时：

```
方案A（最快）: 从视频标题 + 合集目录推断主题
  → 做主题级缺口分析（见§3）

方案B（次快）: 让老板直接提供文案/笔记
  → 拿到后做逐字对比

方案C（完整）: 下载视频→ffmpeg提取音频→whisper转录
  → 耗时较长，仅对高价值视频使用
```

## §3. 缺口分析方法论

### 3.1 主题级缺口分析（无完整文案时使用）

当只能获取视频标题时，按以下流程进行分析：

```
① 提取视频标题中的主题关键词
② 映射到现有技能库的对应章节
③ 检查覆盖率:
   · 关键词命中对应技能 > 5次 → ✅ 已覆盖
   · 关键词命中1-5次 → 🟡 部分覆盖（需进一步检查）
   · 关键词命中0次 → 🔴 缺失
④ 对🟡和🔴的topic，进一步查看技能对应章节的深度
⑤ 输出缺口分析报告
```

### 3.2 全量对比报告模板

```markdown
| 视频主题 | 覆盖率 | 对应技能位置 | 备注 |
|:---------|:-------|:-------------|:-----|
| 主题A | ✅/🟡/🔴 | 技能名+行数 | 说明 |

统计: 已覆盖=N 部分覆盖=N 未覆盖=N

需要补充的缺口（按优先级）:
P0: ...
P1: ...
P2: ...
```

### 3.3 融合标准（原则C的具体化）

新概念融合前必须做以下交叉验证：

```yaml
检查① [v2]: 新概念与体系来源的关系判断？
  ├─ 同一体系内（如泉师兄内部另一篇 vs 已有泉师兄规则）→ 对比逻辑一致性
  │   ├─ 矛盾 → 以已融合版本为准，标记需澄清
  │   └─ 一致或补充 → 继续检查②
  ├─ 不同体系间（如泉师兄盲派 vs 九龙道长传统）→ 判断冲突等级
  │   ├─ 🔴高度冲突（优先级/底层概念相反）→ 不强制融合，保留为独立体系，在技能中标注「盲派专项规则/传统规则」
  │   ├─ 🟡中度差异（方法论不同但可共存）→ 保留两种方法，标注差异性
  │   └─ ✅一致 → 继续检查②
  └─ 无来源概念（纯新理论）→ 进入检查②

检查②: 新概念是否已被现有体系更精细地覆盖？
  ├─ 是 → 不录入（重复）
  └─ 否 → 进入检查③

检查③: 新概念是否只是命名不同但逻辑一致？
  ├─ 是 → 记录命名到对应章节作为"亦称"
  └─ 否 → 作为新增规则录入
```

**重要**：不同体系之间的合法差异（如盲派的「三合>三会」与传统「三会>三合」冲突）是非孰对孰错的理论分歧，不是错误。必须同时保留两种规则并标注所属体系，不得以一者覆盖另一者。

## §6. 全量文案逐字精读 + 系统化融合（2026-07-13新增·泉师兄完整文案萃取）

> **适用场景**：拿到了完整口播文案/课程笔记/逐字稿，需要全量精读并与现有体系深度融合
> **与§3主题级分析的区别**：§3用于只有标题/合集目录时快速扫描缺口；§6用于有完整文案时深度吸收

### 6.1 全量精读七步法

```yaml
Step ① — 逐字精读 + 概念提取
  从头到尾完整阅读全文，不跳读
  对每个概念做标记：已覆盖/部分覆盖/缺失
  特别注意：
    ├─ 核心概念（如本文的"做功""反局""体用"）
    ├─ 比喻/故事（如"格局=地基""体用=战争"）
    └─ 实战案例（如泉师兄列举的具体八字案例）

Step ② — 全量交叉验证（逐条对比现有技能库）
  逐个检查标记的概念：
  ├─ 加载对应技能（skill_view）：foundation → fortune → destiny → wealth → liunian
  ├─ 搜索该概念在当前技能中的出现次数和深度
  └─ 判定：✅已覆盖 / 🟡部分覆盖 / 🔴缺失

Step ③ — 缺口优先级分级
  P0: 核心理论缺口 → 整个体系缺失的重大概念
  P1: 重要应用缺口 → 已有框架但缺少关键细节
  P2: 小众/特殊话题 → 锦上添花

Step ④ — 内容准备（批量起草）
  同时准备所有新增内容的完整草稿：
  ├─ 每个内容独立保存到临时文件（如/tmp/zuogong-content.md）
  ├─ 统一风格：yaml代码块+结构化+口诀结尾
  └─ 确保每个概念：
      · 有明确定义
      · 有判定条件（可操作、可执行）
      · 有实战案例（优先用原文案例）
      · 有口诀助记

Step ⑤ — 插入位置确认（并行子任务）
  用delegate_task派子任务确认每个插入点：
  ├─ 每个子任务只负责一个文件
  ├─ 子任务确认具体行号+前后上下文
  └─ 继续工作不等候（子任务返回时自动进入对话）
```

## §7. 批量文档并行处理工作流（2026-07-13新增·18文档并行萃取实战验证）

> **适用场景**：一次收到多个文档（抖音文案/公众号文章/课程笔记等），需要全量吸收
> **核心**：分组并行 + 逐条交叉验证 + 只提一次也必录

### 7.1 工作流总览

```yaml
收到N个文档 → 以下6步并行处理：

Step ① — 批量提取原始文案
  ├─ .docx → python-docx逐文件提取段落文本
  ├─ .txt  → 直接读取
  ├─ .md   → 直接读取
  └─ 验证：每个文件必须正确提取完整内容（对比字符数）

Step ② — 按主题分组（推荐4组）
  分组原则：每个组内的文档主题相近，交叉验证效率最高
  
  推荐分组：
  ├─ 核心体系组（学习指南/体用/做功/喜用神/大运流年等）
  ├─ 婚姻子女组（婚姻/离异/子女/婚姻总论等）
  ├─ 财富事业组（有钱人/择业/食伤生财/劫财破财/财运等）
  └─ 识人特殊组（识人/小三/同性恋/枭神夺食等）

Step ③ — 并行派发子任务（delegate_task × N组）
  每组用一个delegate_task，prompt包含：
  ├─ 该组的所有文档完整文案
  ├─ 现有技能体系的简要描述（供交叉验证参考）
  ├─ 明确的输出格式要求（见§7.2）
  └─ 特别强调的检查重点（如"只提一次也必录"）

Step ④ — 整合子任务结果
  子任务返回后，合并所有知识点：
  ├─ 按覆盖状态分组（✅已有/🟡部分/🔴缺失）
  ├─ 去重（跨组可能出现重复知识点）
  └─ 输出全量缺口清单（按P0/P1/P2优先级排序）

Step ⑤ — 批量起草+确认位置+执行插入
  ├─ 所有新增内容先准备好完整草稿
  ├─ 用patch工具一次性连续插入
  └─ 已在多文档中被标记为✅的做最终确认

Step ⑥ — 更新萃取记录+推库
  ├─ 在§4.x添加本次萃取的完整记录
  ├─ git commit + push
  └─ commit message注明新增知识点的领域范围
```

### 7.4 批处理质量门（QA Gate）— 提取后强制审计

在Step ④整合子任务结果后，必须执行以下质量检查：

```yaml
QA Gate - 批提取质量门（6道检查）:

  [QA-1] 覆盖率一致性
    ├─ 所有文档是否都被至少一个组覆盖？
    ├─ 是否有漏网文档没有被分配到任何组？
    ├─ 是否有文档在之前单独的会话中已处理过但未纳入统一审计？
    └─ 如果有 → 必须新建独立审计或在报告中注明已单独处理

  [QA-2] 跨文档去重
    ├─ 同一个知识点在多个文档中出现 → 保留一次标注多处来源
    ├─ 如寻庚基同时出现于多个文档 → 去重后只录一次
    └─ 最终输出净知识点数而非原始标记总数

  [QA-3] 只提一次通用规则标记
    ├─ 各文档中只出现一次但属于通用规则的知识点必须单独标注
    ├─ 应在每个组分别执行不能仅限某一组
    └─ 标记格式：⭐1、⭐2... 附重要性等级

  [QA-4] 原始文件归档
    ├─ 提取完成后原始文件必须保留不可删除
    └─ 若已丢失 → 在审计报告中标注无法原文验证

  [QA-5] 体系冲突标记
    ├─ 提取源属于非九龙体系如泉师兄盲派→必须做跨体系交叉验证
    ├─ 对每个新知识点标记：同体系/跨体系
    └─ 跨体系冲突 → 按冲突等级分类标注

  [QA-6] 先前会话知识完整性检查（2026-07-13新增）
    ├─ 检查之前会话是否已处理过同一来源文档但未纳入本次统一审计
    ├─ 如doc1看八字的思路已在之前会话处理但不在本次4组报告中
    ├─ 必须在审计报告中明确标注已单独处理并确认在体系中
    └─ 如无法确认→需重新精读该文档原文确保100%覆盖
```

## §8. 跨体系交叉验证方法论（2026-07-13新增·泉师兄盲派vs九龙道长传统体系）

> **适用场景**：当萃取来源属于非九龙道长的另一套理论体系（如泉师兄盲派、钟义明、段建业等），需要判断其与现有金鉴真人体系（基座=九龙道长）的关系

### 8.1 跨体系交叉验证的核心问题

金鉴真人的体系基座是九龙道长传统理论+能量倍数法。但当淬取来源是泉师兄（盲派）时，两套体系在以下维度可能不同：

| 维度 | 同一体系内验证 | 跨体系验证 |
|:-----|:-------------|:----------|
| 冲突性质 | 逻辑不一致 → 需修复 | 理论分歧 → 需共存 |
| 处理方式 | 融合/覆盖 | 独立标注+归因 |
| 判断标准 | 以体系自身一致性为准 | 以冲突等级分类为准 |
| 目标 | 使得体系自洽 | 使得体系包容不同流派 |

### 8.2 跨体系冲突等级判定

```yaml
🔴 高度冲突 — 根本对立，无法融合
  特征：同一概念有完全相反的定义
  处理：必须在技能中「独立标注」，两套规则分章节保存
        严禁合并或强行统一
  示例：
  - 刑冲合害优先级：泉师兄说三合>三会，九龙说三会>三合 → 分列
  - 体用概念：泉师兄的"体=印比食伤"vs九龙"体=五行本质" → 分列
  - 地支特殊生克：泉师兄说寅辰不论克 → 盲派专项规则

🟡 中度差异 — 方法论不同，可互补共存
  特征：同一问题有不同解法，但理念不根本对立
  处理：两种方法各自保留，在技能中标注差异性和适用场景
        可以用泉师兄方法作为九龙的「升级版/补充版」
  示例：
  - 行业选择：九龙说喜用神匹配行业，泉师兄说需要前置条件
  - 定灾：九龙用倍数法，泉师兄用体受伤定性法

✅ 一致可融合
  特征：逻辑一致或被已有体系更精细覆盖
  处理：直接融合或去重
  示例：家里家外理论、墓库理论
```

### 8.3 跨体系验证流程

```yaml
Step ① — 标识体系来源
  每次萃取前明确标注来源体系：
  ├─ 九龙道长（传统+倍数法）— 金鉴体系基座
  ├─ 泉师兄（盲派·做功体系）
  ├─ 其他（标注流派名称）
  └─ 未知体系 → 先做体系识别

Step ② — 逐条判定冲突等级
  对每个知识点：
  ├─ 同体系内 → 按§3.3检查逻辑一致性
  └─ 跨体系间 → 按§8.2判定冲突等级

Step ③ — 执行对应操作
  🔴高度冲突 → 独立章节标注「流派差异」
  🟡中度差异 → 在对应章节补充「另见：XX流派法」
  ✅一致      → 直接融合

Step ④ — 记录冲突矩阵
  每完成一次跨体系萃取，更新冲突矩阵表（见references/参考文件）：
  ├─ 冲突名
  ├─ 两体系各自观点
  ├─ 冲突等级
  └─ 处理决定
```

## §9. 静态HTML网页/在线书籍提取方法（2026-07-14新增·算准网盲派初级命理学实战验证）

> **适用场景**：从静态HTML书网站（如算准网 suanzhun.net、国学典籍网站等）提取完整书籍/章节内容
> **技术特点**：无需浏览器/Playwright，纯curl+sed即可完成多页HTML内容提取和结构化输出
> **补充关系**：§2用于抖音视频，§6用于文字文案，§9用于HTML静态网页书籍

### 9.1 完整提取流水线

```yaml
Step ① — 确定总页数
  ┌─ curl获取第一页HTML
  ├─ 检查分页导航（页码链接通常为 /book/{id}.html, /book/{id}_2.html, ...）
  ├─ 常见模式：页码在页面底部 <a title="Page"> <b>1</b>/<b>8</b> </a>
  └─ 确认总页数N后，准备N个并行请求

Step ② — 批量并行提取原始HTML（curl + sed内容截取）
  提取命令：
    curl -s 'https://www.suanzhun.net/book/{id}_N.html' \
    | sed -n '/<div class="content-text">/,/<\/div>/p'

  关键：sed范围匹配 '<div class="content-text">' 到 '</div>'，这是大多数书网站的通用模式

Step ③ — HTML标签清洗 + 数据结构化
  清洗流水线（单行sed链）：
    sed 's/<br \/>/\n/g;                      # 换行标签→换行符
         s/<[^>]*>//g;                         # 移除全部HTML标签
         s/&nbsp;//g;                           # 空格实体
         s/&ldquo;/"/g; s/&rdquo;/"/g;          # 中文引号
         s/&lt;/</g; s/&gt;/>/g;               # 尖括号
         s/&amp;/\\&/g;                         # &符号
         s/^ *//; s/ *$//'                     # 首尾空白

Step ④ — Page N合并 + 内容校验
  ┌─ 逐页合并为完整章节文本
  ├─ 检查内容连续性（上一页末尾与下一页开头应无缝衔接）
  └─ 确认无缺失页码后，进入知识提取阶段

Step ⑤ — 知识提取 + 结构化输出
  建议格式（三列知识表）：
    | 知识点 | 原文 | 章节 |
    优势：原文可追溯、分类清晰、便于后续对比融合
```

### 9.2 多页面的URL模式识别

| 页面模式 | URL格式 | 示例 |
|:--------|:--------|:-----|
| 第1页 | `{base}.html` | `/book/1249.html` |
| 第N页 | `{base}_{N}.html` | `/book/1249_2.html` 到 `/book/1249_8.html` |
| 末页 | `{base}_{N}.html` | `/book/1249_8.html` |

**注意**：`{N}` 不一定是第一页——验证方法：检查第1页HTML中的分页导航元素。

### 9.3 内容清洗注意事项

```yaml
常见HTML实体转换表：
  原始实体 | 含义    | 应替换为
  &ldquo;   | 左引号  | "
  &rdquo;   | 右引号  | "
  &nbsp;    | 空格    | (移除)
  &lt;      | <       | <
  &gt;      | >       | >
  &amp;     | &       | &

结构元素处理：
  <br \/>     → 换行 (保留行结构)
  <p>...</p>  → 段前加空行
  &nbsp;      → 直接移除（非中文空格在提取中无意义）

分页前后衔接处理：
  ┌─ 第i页末尾可能不完整（句子/段落被截断）
  ├─ 第i+1页开头也可能不完整
  └─ 合并后应逐段还原为完整段落
```

### 9.4 输出格式选择建议

```
情景A — 低深度：只需快速扫描 + 对比
  格式：知识表（| 知识点 | 原文 | 章节 |）
  优势：每个知识点直接可追溯原文，适合快速确认覆盖状态
  劣势：上下文略有割裂

情景B — 高深度：需要完整理解体系
  格式：原文分段 + 知识点标记（在原文中标注 🔴🟡✅）
  优势：保留完整语境，适合逐字精读
  劣势：篇幅较大，后期整理需要额外步骤

情景C — 知识库归档：长期保存
  格式：纯markdown章节结构（保留原书的§层级）
  优势：可直接作为知识库永久文档
  劣势：需要额外排版工作
```

### 9.5 实战禁忌

```yaml
❌ 不要只提取第一页就以为完整——确认总页数后再开始批量提取
❌ 不要在不知道总页数时逐个递增URL尝试（效率低且可能触发反爬）
❌ 不要对清洗后的文本再加工导致原文失真（保留原文用于追溯验证）
✅ 先取第1页看分页导航 -> 计算总页数 -> 全部并行提取 -> 合并清洗
✅ 提取时同时保留原始HTML和清洗后文本（原始HTML用于排查清洗异常）
```

### 9.6 实战口诀

```yaml
一看分页定总数，二并行curl全拉走
三用sed清洗标签，&ldquo;&rdquo转引号
四合并页要检查，头尾衔接无断层
五提知识结构化，三列表格留原文
情景A快速对比，情景B深度精读
原始HTML先备份，清洗异常可溯源
```
