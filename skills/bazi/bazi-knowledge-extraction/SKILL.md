## §2. 抖音视频提取技术（方法栈 · 2026-07-10验证）

### 2.0 完整方法栈（按优先级排列）

| 优先级 | 方法 | 状态 | 说明 |
|:------|:-----|:-----|:------|
| ① | Playwright浏览器自动化 | ⚠️ 可行 | 可获标题+描述+合集目录，点"展开"按钮获取更多内容。用手机设备模拟(iPhone UA)效果更好 |
| ② | **第三方转录工具** (anxinxia等) | ⚠️ 需非黑名单IP | `jiexi.anxinxia.com/tiqu` 提供在线视频文案提取(ASR语音转文字)。工作流：解析视频→下载音频→转文字。该工具本身使用faster-whisper/类似ASR引擎，需从非黑名单IP调用(服务器IP被抖音屏蔽时也会失败) |
| ③ | yt-dlp + 登录态cookies | ⚠️ 需cookies | 有登录态cookies后可下载音频→Whisper转录得到完整口播文案。faster-whisper已安装可用 |
| ④ | 第三方解析API | ❌ 接口过期 | `douyin` python包(2022) amemv.com接口已失效 |
| ⑤ | 抖音REST API / amemv.com内部API | ❌ 需签名参数 | 需要X-Bogus签名+app版本参数，黑名单IP即使有签名也被拦截 |

**核心结论**: 
- 获取完整口播文案的完整流水线: `获取视频URL → 解析下载地址 → ffmpeg提取音频 → faster-whisper转文字`
- 关键瓶颈: 服务器IP若被抖音列入黑名单，解析下载地址这一步在所有方法上都会失败
- 解决方案A: 用非黑名单的服务器做中转
- 解决方案B: 让老板直接从手机端使用anxinxia类工具提取粘贴
- 无cookies/非黑名单IP时，只能用Playwright获取标题/描述/合集，做主题级缺口分析

### 3.3 融合标准（原则C的具体化）

新概念融合前必须做以下交叉验证：

```yaml
检查① [v2]: 新概念与体系来源的关系判断？
  ├─ 同一体系内（如泉师兄内部另一篇 vs 已有泉师兄规则）→ 对比逻辑一致性
  │   ├─ 矛盾 → 以已融合版本为准，标记需澄清
  │   └─ 一致或补充 → 继续检查②
  ├─ 不同体系间（如泉师兄盲派 vs 九龙道长传统）→ 判断冲突等级
  │   ├─ 🔴高度冲突（优先级/底层概念相反）→ 不强制融合，保留为独立体系，在技能中标注「盲派专项规则/传统规则」
  │   ├─ 🟡中度差异（方法论不同但可共存）→ 保留两种方法，标注差异性
  │   └─ ✅一致 → 继续检查②
  └─ 无来源概念（纯新理论）→ 进入检查②

检查②: 新概念是否已被现有体系更精细地覆盖？
  ├─ 是 → 不录入（重复）
  └─ 否 → 进入检查③

检查③: 新概念是否只是命名不同但逻辑一致？
  ├─ 是 → 记录命名到对应章节作为"亦称"
  └─ 否 → 作为新增规则录入
```

**重要**：不同体系之间的合法差异（如盲派的「三合>三会」与传统「三会>三合」冲突）是非孰对孰错的理论分歧，不是错误。必须同时保留两种规则并标注所属体系，不得以一者覆盖另一者。

### 7.1 工作流总览

```yaml

  ├─ .docx → python-docx逐文件提取段落文本
  ├─ .txt  → 直接读取
  ├─ .md   → 直接读取
  └─ 验证：每个文件必须正确提取完整内容（对比字符数）

Step ② — 按主题分组（推荐4组）
  分组原则：每个组内的文档主题相近，交叉验证效率最高
  
  推荐分组：
  ├─ 核心体系组（学习指南/体用/做功/喜用神/大运流年等）

  ├─ 明确的输出格式要求（见§7.2）
  └─ 特别强调的检查重点（如"只提一次也必录"）

Step ④ — 整合子任务结果
  子任务返回后，合并所有知识点：
  ├─ 按覆盖状态分组（✅已有/🟡部分/🔴缺失）
  ├─ 去重（跨组可能出现重复知识点）
  └─ 输出全量缺口清单（按P0/P1/P2优先级排序）

  ├─ 在§4.x添加本次萃取的完整记录

## §8. 跨体系交叉验证方法论（2026-07-13新增·泉师兄盲派vs九龙道长传统体系）

### 🔴 实战陷阱：混淆体系边界（2026-07-14 两次校准触发）

> **致命错误**：将「象法」归入「盲派」体系。老板纠正：「象法和盲派应该是两个体系吧？你把它们并列出来」
> **根本原因**：泉师兄同时讲盲派（做功/体用/归属权）和象法（干支象形/取象/画图），两者在视频中交替出现→被我误认为同一体系

```yaml
体系边界判定检查清单（每次萃取前确认）：

① 来源中涉及多个体系吗？
   └─ 如果是 → 必须分别列出每个体系，不得合并

② 每个体系的范围是什么？
   └─ 盲派（做功/体用/宾主/归属权/家里家外/寻根基）

   └─ 九龙道长（身强弱/喜用/格局/倍数法）

③ 体系之间有交叉吗？
   └─ 有交叉是正常的（象法可以辅助盲派判断行业方向）
   └─ 交叉≠合并——保留独立体系，只做「交叉引用」

④ 命名是否正确？
   └─ 「象法」不是「大象方法论」——象法是象形法（象形文字），读的是形象/形状

处理原则：
  多体系并存时 → 每个体系独立成章
  交叉引用时 → 注明「交叉引用：XX体系」
  模糊边界时 → 先按独立体系构建，再补交叉引用

### 8.2 跨体系冲突等级判定

```yaml
🔴 高度冲突 — 根本对立，无法融合
  特征：同一概念有完全相反的定义
  处理：必须在技能中「独立标注」，两套规则分章节保存

  - 地支特殊生克：泉师兄说寅辰不论克 → 盲派专项规则

🟡 中度差异 — 方法论不同，可互补共存
  特征：同一问题有不同解法，但理念不根本对立
  处理：两种方法各自保留，在技能中标注差异性和适用场景

  示例：
  - 行业选择：九龙说喜用神匹配行业，泉师兄说需要前置条件
  - 定灾：九龙用倍数法，泉师兄用体受伤定性法

✅ 一致可融合
  特征：逻辑一致或被已有体系更精细覆盖
  处理：直接融合或去重
  示例：家里家外理论、墓库理论
```

## §9. 静态HTML网页/在线书籍提取方法（2026-07-14新增·算准网盲派初级命理学实战验证）

> **适用场景**：从静态HTML书网站（如算准网 suanzhun.net、国学典籍网站等）提取完整书籍/章节内容
> **技术特点**：无需浏览器/Playwright，纯curl+sed即可完成多页HTML内容提取和结构化输出
> **补充关系**：§2用于抖音视频，§6用于文字文案，§9用于HTML静态网页书籍

### 9.1 完整提取流水线

```yaml
Step ① — 确定总页数
  ┌─ curl获取第一页HTML
  ├─ 检查分页导航（页码链接通常为 /book/{id}.html, /book/{id}_2.html, ...）
  ├─ 常见模式：页码在页面底部 <a title="Page"> <b>1</b>/<b>8</b> </a>
  └─ 确认总页数N后，准备N个并行请求

Step ② — 批量并行提取原始HTML（curl + sed内容截取）
  提取命令：
    curl -s 'https://www.suanzhun.net/book/{id}_N.html' \
    | sed -n '/<div class="content-text">/,/<\/div>/p'

  关键：sed范围匹配 '<div class="content-text">' 到 '</div>'，这是大多数书网站的通用模式

Step ③ — HTML标签清洗 + 数据结构化
  清洗流水线（单行sed链）：
    sed 's/<br \/>/\n/g;                      # 换行标签→换行符
         s/<[^>]*>//g;                         # 移除全部HTML标签
         s/&nbsp;//g;                           # 空格实体
         s/&ldquo;/"/g; s/&rdquo;/"/g;          # 中文引号
         s/&lt;/</g; s/&gt;/>/g;               # 尖括号
         s/&amp;/\\&/g;                         # &符号
         s/^ *//; s/ *$//'                     # 首尾空白

Step ④ — Page N合并 + 内容校验
  ┌─ 逐页合并为完整章节文本
  ├─ 检查内容连续性（上一页末尾与下一页开头应无缝衔接）
  └─ 确认无缺失页码后，进入知识提取阶段

Step ⑤ — 知识提取 + 结构化输出
  建议格式（三列知识表）：
    | 知识点 | 原文 | 章节 |
    优势：原文可追溯、分类清晰、便于后续对比融合
```

### 9.2 多页面的URL模式识别

| 页面模式 | URL格式 | 示例 |
|:--------|:--------|:-----|
| 第1页 | `{base}.html` | `/book/1249.html` |
| 第N页 | `{base}_{N}.html` | `/book/1249_2.html` 到 `/book/1249_8.html` |
| 末页 | `{base}_{N}.html` | `/book/1249_8.html` |

**注意**：`{N}` 不一定是第一页——验证方法：检查第1页HTML中的分页导航元素。

### 9.3 内容清洗注意事项

```yaml
常见HTML实体转换表：
  原始实体 | 含义    | 应替换为
  &ldquo;   | 左引号  | "

  &nbsp;    | 空格    | (移除)
  &lt;      | <       | <
  &gt;      | >       | >
  &amp;     | &       | &

结构元素处理：
  <br \/>     → 换行 (保留行结构)
  <p>...</p>  → 段前加空行

分页前后衔接处理：
  ┌─ 第i页末尾可能不完整（句子/段落被截断）
  ├─ 第i+1页开头也可能不完整
  └─ 合并后应逐段还原为完整段落

## §10. 动态中文文章页提取法（2026-07-14新增·神机阁/国学类网站实战验证）

> **适用场景**：从动态渲染的中文文章/博客网站（如 shenjige.cn、国学类、命理类等静态内容网站）提取完整长文
> **与§9的区别**：§9针对**多页静态HTML书籍**（curl+sed），§10针对**单页动态渲染长文章**（浏览器JS直接提取全文本）
> **典型痛点**：浏览器加载成功但 browser_snapshot 只能获取前 ~5000字符，后面内容被截断

### 10.1 问题识别

| 现象 | 判断标准 | 应采取的方法 |
|:-----|:---------|:------------|
| browser_navigate 成功，snapshot 有完整标题但正文被截断 | 控制台输出 "[... N more lines truncated]" | §10 方法 |
| browser_snapshot 包含 heading 但 paragraph 内容是空的详情感叹号 | 页面靠 JS 动态渲染，snapshot 抓不到完整文本 | §10 方法 |
| 页面完全加载但 snapshot 只显示前几个段落 | 单页长内容，非分页结构 | §10 方法 |
| 页面包含分页导航（页码 1/8） | 多页书籍结构 | §9 方法 (curl+sed) |
| 页面包含 Cloudflare/验证码 | 无法提取 | 放弃并换源 |

### 10.2 提取方法栈（按优先级）

```yaml
方法① — browser_console + JS提取（推荐）
  原理：已加载的页面 DOM 包含全部文本，只是 snapshot 截断了。用 JS 直接读取 DOM 可获取全文。
  
  基本命令：
    document.querySelector('article').innerText
    或 document.querySelector('.content').innerText
    或 document.body.innerText（兜底）
  
  执行：
    browser_console(expression="document.querySelector('article').innerText")
  
  注意事项：
    ├─ 优先尝试 'article' 标签（语义标签，大多数中文文章站使用）
    ├─ 其次尝试 '.content' / '.article-content' / '#content' 类选择器
    ├─ 最后用 'document.body.innerText' 兜底（会包含导航等额外内容）
    └─ result 返回完整文本（可能很长，但一条 JS 语句即可获取全部）

方法② — 滚动 + 再次 snapshot（辅助方法）
  当方法①不可用时（页面加载不全），先 browser_scroll(direction='down') 加载更多内容
  再配合方法①提取

方法③ — curl + 分析 HTML 结构
  如果页面不依赖 JS 渲染：curl + grep 匹配文章内容标签
  适用场景：页面是服务端渲染的纯 HTML（多数中文文章站如此）
  
  命令：
    curl -sL 'URL' -A 'Mozilla/5.0 ...' | grep -oP '(?<=<article>).*(?=</article>)' | head -c 5000
```

### 10.3 实战步骤（标准流程）

```yaml
Step ① — browser_navigate 加载页面
  └─ 检查 snapshot：标题、摘要、主要段落是否完整
     ├─ 标题缺失 → 方法③（curl直取）
     ├─ 内容完整 → 直接使用 snapshot 或 browser_snapshot(full=true)
     └─ 内容截断 →
        Step ② — 用 browser_console 执行 JS 提取全文
          └─ 重点：检查 result_type 是否为 'str'，result 是否包含完整文本
        Step ③ — 如 method① 不足（动态加载内容）：
          browser_scroll(direction='down')
          browser_console 重试
        Step ④ — 提取成功 → 保存到文件

常见问题处理：
  ├─ document.querySelector('article') 为空 → 试试 'main' / '.post-content' / '#content' / '.article-body'
  ├─ 提取的文本包含页眉页脚导航 → 用 .replace() 修剪
  ├─ 文本中含有 HTML 实体（&nbsp;等）→ 终端 sed 清洗后保存
  └─ 返回为 {} 或空字符串 → 弃用 browser_console，改用 curl + 正则
```

### 10.4 实战验证：神机阁 (shenjige.cn) 全文提取

> **日期**：2026-07-14
> **URL**：https://www.shenjige.cn/details/dkJSNhapY.html

### 🚨 老板铁律：去伪第一，绝不污染知识库（2026-07-14 本条会话老板校准）

> **直接引用的老板原话**：「别把错误的东西沉淀下来，污染知识库」
> **规则**：知识萃取的第一目标不是「能录多少」，而是「不把错的装进来」

```yaml
底层原则：宁可少录，不可错录
  错误的规则一旦入库，会在后续所有八字分析中被反复引用→污染成倍放大
  正确率优先级：⭕ 覆盖率 < ⭕ 交叉验证 < ⭕ 零错误

强制执行的点：
① 每条知识点必须有至少2个独立来源验证 → 非「该来源说的有道理就行」
② 只有1个来源但该来源权威且无任何矛盾 → 标注「仅来源X·待后续验证」
③ 有矛盾的（相同概念不同说法）→ 不合并不硬融
   ├─ 跨体系矛盾 → 标注「流派差异·各自独立」
   └─ 同体系矛盾 → 以体系内的确认为准，矛盾方不入库
④ 自己推理出来的结论 → 必须标注「推理结论」，不能在应用中当确定规则用
⑤ 不确定的 → 不录，标记「待老板确认」

质量门（每一步都必须问自己）：
  ┌─ 这条知识我能追溯原文吗？                 → 能追溯才录
  ├─ 这条知识至少有2个来源吗？                 → 有才录
  ├─ 这条知识和已有体系有矛盾吗？              → 有矛盾不融合只并列
  ├─ 这条知识是我自己推理的还是从原文提取的？  → 推理的标注清楚
  └─ 这条知识哪怕有问题，污染面有多大？        → 宁可不录
```

### 🔴 致命陷阱：并发OCR进程内存争抢

> **教训**：同时启动6个PDF OCR作业（每个12页·300DPI），5分钟后全部卡住（CPU 0.8%不动）。改为顺序执行后，每个12页PDF约60-90秒完成。
>
> **根因**：pdf2image将整个PDF一次性加载到内存（12页×300DPI PNG ≈ 24-36MB/进程），并发5个时总内存超限，进程开始Swap→卡死→无进度。
>
> **铁律**：扫描件PDF OCR**必须顺序执行**，一次只跑一个文件。

```yaml
正确执行模式：
  ❌ python3 ocr_pdf.py 二 ... &  # 后台并发 → 卡死
  ❌ python3 ocr_pdf.py 三 ... &  # 后台并发 → 卡死
  ✅ python3 ocr_pdf.py 二 ...    # 顺序执行第1个（~80秒）
  ✅ python3 ocr_pdf.py 三 ...    # 上一步完成后执行第2个（~80秒）
```

  加载所有相关技能 → 逐条比对：
  ├─ ✅ 已覆盖 → 不处理
  ├─ 🟡 部分覆盖 → 准备补充内容

  ├─ P0: 核心理论缺口（立即补）
  ├─ P1: 重要应用缺口（排队补）
  ├─ P2: 附录性质（可新建轻量技能）
  └─ SOP同步更新（如有新增流程/步骤）

  更新方式：
  ├─ 现有技能补充：patch末尾追加
  ├─ 全新技能：skill_manage create
  ├─ 自动化体系：patch SOP
  └─ 并行更新时注意：delegate_task派发（每个任务2个技能）

Step ⑦ — 双库推库
  ├─ 知识库：git add + commit + push
  ├─ 技能库：git add + commit + push
  ├─ 清理：删除临时OCR文件（cache目录）
  └─ 汇报：给老板最终处理统计
```

### 13.3 致命陷阱：OCR术语不可直接使用，必须经域知识验证

> 2026-07-14 老板两次纠正触发：第一次我把OCR识别的「芝位」直接用了，老板纠正「不是芝位体系，是八卦爻位体系」。第二次修正时又漏了多处，老板说「所有错的地方全部改完，都是关于八卦八字命理的」。

**铁律**：OCR输出的命理术语**不可直接使用**，必须经过三重验证：
1. **域知识检查**——问自己：「八卦命理中有这个术语吗？」（如「芝位」不存在，正确是「爻位」）
2. **原始图片确认**——用 vision_analyze 看PDF截图中的原文
3. **老板校准**——不确定时宁可先标记「待确认」也不要直接用OCR原文

### 致命陷阱2：OCR修正必须全库扫描，不能只修表面（2026-07-16 老板校准触发）

> **教训**：修正「芝位→爻位」后以为改完了，老板说「所有错的地方全部改完」→ grep 发现还有多处残留（功位/攻位/区位等OCR变体）。再搜「印组」发现技能文件中也有。

**铁律**：OCR修正必须执行「三点三面扫描法」：

```yaml
第1点 - 点（当前文件）
  把当前文件内所有该术语的变体全部搜出来替换掉

第2点 - 面（同仓库所有文件）
  grep 全仓库搜索该术语的所有变体
  不仅搜索错误词本身，还要搜索可能存在的其他OCR变体
  例：搜「芝位」也要搜「功位」「攻位」「区位」「印组」「天王」「俯伤」

第3点 - 体（技能库+知识库双库）
  知识库 grep → 修正
  技能库 grep → 修正
  最后再次全库 grep → 确认零残留

执行流程：
  ① 建立修正映射表（错误词→正确词）
  ② 在修正映射表中追加本次发现的新变体
  ③ 对每个变体执行：知识库grep→修正→技能库grep→修正
  ④ 全部修完后：全库再次grep→确认0残留
  ⑤ 更新 ocr-correction-table.md 参考文件
```

### 致命陷阱3：新会话可能跳过时间验证（2026-07-16 老板校准触发）

> **教训**：SOUL.md有「每次会话自动执行 time verification」的规则，但上下文压缩后新会话开始时没有执行。老板指出后才意识到跳过了。
>
> **根因**：上下文压缩后，前一个session的末尾成了新session的开头。SOUL.md的规则虽然在同一次会话中持续存在，但上下文压缩+时间戳残留导致我以为日期是对的。

**铁律**：每次会话的第一条消息发出前，必须执行 `terminal('date')` 确认服务器时间。已焊入SOUL.md：

```yaml
强制流程：
  Step ① - 收到用户消息后，第一条回复必须先执行 terminal('date')
  Step ② - 确认服务器时间与预期一致（北京时间 CST/UTC+8）
  Step ③ - 如果不一致，以服务器时间校准所有时间相关的判断
  Step ④ - 校准后再开始正式回复
  - 禁止：不执行date就开始写分析/报告
  - 禁止：依赖Hermes对话中的时间戳（可能被上下文压缩残留污染）
```
