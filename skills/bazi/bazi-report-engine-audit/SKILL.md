## 第一部分：引擎深度报告 vs 手写报告的区别

引擎深度报告（由 `generate_deep_report.py` 生成）和手写21§报告在深度上有本质区别：

| 维度 | 引擎深度报告 | 手写21§报告 |
|:-----|:------------|:------------|
| 生成方式 | `generate_deep_report(pipeline的结果)` | 逐§按skill规则手动写 |
| §深度 | 引擎detail_analysis决定（每§几十行） | 每§几百行 |
| §6性格 | 35~40行 | ≥120行 |
| §8财富 | 50~60行 | ≥80行 |
| §11学历 | 40~45行 | ≥60行 |
| §12婚姻 | 25~30行 | ≥56行 |
| §14健康 | 25~30行 | ≥50行 |
| §16事件 | 引擎输出+自动生成 ≥70 ✅ | 手动 ≥70行 |
| 数据准确性 | ✅ 引擎规则计算 | ⚠️ 人工易出错 |
| 维护成本 | 低（修引擎一次，所有报告生效） | 高（每份逐§写） |

**核心原则**：引擎深度报告保证数据准确性和一致性，深度受限于引擎输出量。
如果需要更高深度，要扩展引擎模块的 `detail_analysis` 生成逻辑，不是在report层拼凑。

## 第三部分：generate_deep_report.py 常见机制漏洞

### 漏洞1：版本号硬编码
- **症状**：文件名v26.2但内部写v4.0
- **根因**：`generate_deep_report()` 硬编码 `f"...v4.0..."` 
- **修复**：添加 `version` 参数，调用时传入
```python
def generate_deep_report(engine_json, name="", gender="", version="1.0"):
    lines.append(f"# {name}...v{version}...)
```
- **调用链修复**：`report_generator.py` 调用时也要传 `version`

### 漏洞2：缺失字段读取路径
- **症状**：§1中「出生时间」「空亡」「格局条件」「现行大运」为「—」
- **根因**：
  - 日主读取：`bd.get("ri_zhu",{}).get("gan")` 但 `basic_data` 可能是空dict → 加fallback到 `s1.get("ri_zhu",{}).get("gan")`
  - 空亡读取：`s1.get("kong_wang", "—")` 但pipeline_v5没有输出kong_wang到sec_1_overview → pipeline_v5需补充
  - 格局条件读取：`s2.get("condition", "—")` 但pipeline_v5没有输出condition → pipeline_v5需补充
  - 现行大运：用 `15<=start_age<=50` 硬编码判断 → 改为基于当前年份命中
- **修复**：
  1. pipeline_v5的`sec_1_overview`补充：kong_wang, birth_info, 等
  2. pipeline_v5的`sec_2_ge_ju`补充：condition
  3. generate_deep_report的读取路径加双源fallback

### 漏洞3：从弱格判定误用「中和」描述
- **症状**：主母报告§3.2写「从弱（50.0分）——中和...」矛盾
- **根因**：`sqr_n >= 40` 分支包括了从弱(50分)，但用了中和的描述
- **修复**：在 `sqr_n >= 60` 前加 `if "从弱" in str(sqr_label)` 分支

### 漏洞4：白话解读「身身强」重复字
- **症状**：「身身强（64.0分）」——多了一个「身」
- **根因**：`f"身{sqr_label}..."` 但 sqr_label 已经是「身强」
- **修复**：去掉前缀「身」，或判断 sqr_label 是否已以「身」开头

### 漏洞5：五行能量流暴露JSON
- **症状**：`"五行能量分布：{"金": 3, "水": 1, ...}"` 直接输出Python dict
- **根因**：`json.dumps({...})` 直接嵌入字符串
- **修复**：改为自然语言描述如「金3.0分、水1.0分」

### 漏洞6：从弱格白话用身强描述
- **症状**：从弱格写「体质强健、能扛压力」
- **根因**：`sqr_n >= 40` 条件分支，从弱50分落入此分支
- **修复**：独立从弱格白话：「从弱反强，弃命从势，顺势而为不逆势」

### 漏洞7：§13/§14 detail_analysis暴露Python dict
- **症状**：`生育力：{'时支': '卯', '基础等级': '最强', ...}`
- **根因**：引擎的 `detail_analysis` 中某些字段是str(dict)格式
- **修复**：使用 `_dict_to_text()` 函数（见第四部分）

### 漏洞8：§16事件表不足70条
- **症状**：引擎只输出12~15件事件，事件表只有22行
- **根因**：引擎的 `key_events` 只有有限的流年事件
- **修复**：基于大运数据自动生成补充事件（大运开局/中段/末年 + 学业/婚姻/子女特定大运事件）

### 漏洞9：品牌残留
- **症状**：报告中出现「九龙道长」
- **根因**：`generate_deep_report.py` 的法规引用行硬编码
- **修复**：全文搜索 `九龙道长`，替换为「金鉴真人体系（基于原始素材）」

## 第四部分：工具函数 _dict_to_text

用于将引擎detail_analysis中字符串形式的dict转为自然语言文本：

```python
import ast

def _dict_to_text(val, default=""):
    """将可能为dict的值转为自然语言文本（支持字符串形式的dict）"""
    # 尝试解析字符串形式的dict
    if isinstance(val, str) and val.startswith("{"):
        try:
            val = ast.literal_eval(val)
        except (ValueError, SyntaxError):
            pass
    if isinstance(val, dict):
        parts = []
        for k, v in val.items():
            if isinstance(v, (dict, list)):
                # 递归处理子dict
                ...
            else:
                parts.append(f"{k}：{v}")
        return "，".join(parts)
    return str(val) if val else default
```

使用场景：
- §13子女：`生育力={_dict_to_text(sp, '中等')}`
- §13子女：`子女成就={_dict_to_text(ca)}`
- §14健康：`七杀攻身={_dict_to_text(qsd, '无显著信号')}`
- §14健康：`偏印淤堵={_dict_to_text(ppd, '无明显淤堵')}`

### 漏洞15：report_generator.py 传空 paipan/basic_data（2026-07-05发现）
- **症状**：`generate_deep_report()` 的 `pp` 和 `bd` 在整个执行中为空 dict `{}`，导致 §2.3 月令显示「?」、§2.4 五行能量流「暂不可用」

## 第七部分：内容级·九龙道长原始规则对照验证（2026-07-05老板校准）

> 🚨 **致命教训**：2026-07-05老板指出——我只验证了「代码能跑、字段不报错、数据显示了」，**没有拿九龙道长原始理论逐条对比验证每个判断的内容正确性**。
> 这是本质区别。「验证」不只是检查格式/字段/行数，而是：**报告的每个数据点，都必须追溯到引擎计算 AND 九龙道长原始规则，展示完整的证据链。**
>
> 老板原话：**「你每一步验证生成的内容，和相应的九龙道长原始理论体系规则做对比，验证正确与否了吗？就像 debug 一样，深入到每一步细节。」**
>
> **核心要求**：不能只是代码级验证（字段存在、不报错），必须做内容级验证（每个值是否正确、是否符合原始规则）。

### 7.1 证据链验证格式

每次验证报告中的关键数据时，必须输出以下形式的证据链：

```markdown
## 🔬 §X 字段名 · 原始规则对照

| 对照项 | 九龙道长原始规则 | 引擎计算 | 判定 |
|:-------|:----------------|:---------|:----|
| 规则依据 | 原始素材行号/规则名 | 引擎细节值 | ✅/❌ |

子项明细（逐位置/逐项）：
- ① 位置/五行检查 → 规则依据（素材行号） → 计算过程 → 得分 → 结论
- ② 同上...
```

**关键**：每个结论都必须是「规则说X → 引擎算Y → X==Y? ✅/❌」的证据链，不能只说「正确」。

### 7.2 必须做内容级验证的关键模块

### 7.3 验证操作步骤

# 三段式验证链——必须逐项完成：

### 起运年龄错误案例
- **症状**：家主报告起运73.0岁（应为0.3岁），主母72.7岁（应为6.0岁）
- **根因**：农历日期换算错误（农历六月廿五→公历7月20日❌→应为8月6日✅）
- **教训**：农历转公历必须先用 `lunar_to_solar()` 确认，不能凭感觉换算
- **验证方式**：运行引擎后，检查起运年龄是否合理（婴幼儿通常0~6岁，儿童6~12岁）

### 大运排序与引擎数据同步
- 引擎输出的 `best_da_yun` / `worst_da_yun` 基于评分计算
- 但起运年龄改变后，整个大运序列会变（如从6.0→0.3岁起运，第一步大运从甲申变甲午）
- **修复后必须重新运行引擎**，不能基于旧的大运序列做分析
