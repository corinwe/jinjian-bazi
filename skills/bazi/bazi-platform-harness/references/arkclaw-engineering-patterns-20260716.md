## 📐 总体架构（7模块流水线）

```
需求澄清（Step 1）
  ↓
模块拆解（Step 2）：
  ├─ ① 配置校验
  ├─ ② 文件接入
  ├─ ③ PDF解析
  ├─ ④ 指标抽取 ← AI真正干活的地方
  ├─ ⑤ 数据校验 ← 给AI输出加安全网
  ├─ ⑥ 报告生成
  └─ ⑦ 任务调度
  ↓

  ↓

  ↓
多入口设计（Step 5）：同引擎，三种入口覆盖不同使用习惯
```

## 🔥 值得借鉴的 3 个模式

### 模式①：错误隔离屏障（Error Isolation Barrier）

**原文定位**：Step 4「错误处理——不让用户看到技术报错」

| 方面 | ArkClaw做法 | 我方现状 | 差距 |
|:-----|:------------|:---------|:-----|
| **用户端** | 只展示业务层提示，永不暴露stack trace | 引擎JSON解析错误可能透传 | ⚠️ |
| **开发端** | 完整错误日志独立记录，不污染用户界面 | pipeline日志与输出混在一起 | ⚠️ |
| **分级** | 区分「用户可见错误」vs「系统内部错误」 | 未做分级 | ❌ |

**建议改进**：加一个 `error-shield.py` 或 `error-barrier` 模块：

```
输入 → [引擎处理] → [error-barrier] → 用户可见输出
                           ↓
                     [完整错误日志] → 开发者查看
```

规则：
- 所有异常 → 统一文案「加载中，请稍后重试」
- 具体错误信息只写入日志文件，不透传
- HTTP 状态码分层：4xx=用户输入问题, 5xx=系统内部问题, 但消息体不暴露详情

### 模式②：多入口设计（Multi-Entry Architecture）

**原文定位**：Step 5「三种入口，覆盖不同使用习惯」

| 入口 | 场景 | 我方对照 |
|:-----|:------|:---------|
| 飞书Bot | 对话式使用，自然语言输入 | ✅ 已有飞书渠道 |
| 控制台（Web UI） | 可视化操作，参数配置 | ✅ 已有前端SPA |
| API | 程序化调用，集成到其他系统 | ✅ 已有FastAPI |

**启示**：我们的架构已经支持多入口，但需要明确**同一个引擎支撑多种接入方式**的架构原则。目前是自动实现了，但没有主动设计。

---

### 模式③：AI输出安全网（AI Output Safety Net）

**原文定位**：模块5「数据校验——给AI输出加一道安全网」

ArkClaw把数据校验作为一个独立模块（不是嵌入在AI推理步骤中），明确区分：
- 模块4：AI干活（指标抽取）
- 模块5：安全网（校验AI输出）

**对照**：我们已经有 `pillar-verify.py` + `delivery-gate.py` 做校验，但它们是作为「发布前闸门」存在，不是作为「引擎→LLM翻译」之间的中间层。

**建议**：考虑在 pipeline_v5 输出和 LLM 报告生成之间加一层结构校验中间件——在 make report 之前先过结构校验，确保 JSON 字段完整性 + 数值合理性 + 十神一致性。

## ⚖️ 我方已占优的方面

| 维度 | ArkClaw方案 | 我方方案 | 优势方 |
|:-----|:------------|:---------|:-------|
| **计算引擎** | 依赖LLM做指标抽取 | Python引擎做所有计算，LLM只做翻译 | ✅ 我方 |
| **校验体系** | 1道安全网 | 320门禁 + 5关pillar + 交付门禁 | ✅ 我方 |
| **分工体系** | 单体Agent | Sub-Agent 7种分工 + Maker/Checker | ✅ 我方 |
| **知识体系** | 通用AI能力 | 九龙道长+泉师兄双体系，非通用LLM知识 | ✅ 我方 |
