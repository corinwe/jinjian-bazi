# 全量审计方法论（2026-06-14沉淀）

> **核心原则**：审计必须「所有技能 vs 全部理论」，不是「新技能 vs 旧技能」。
> 用户明确纠正：「是对比所有技能包和全部理论知识，不是拆分后的技能跟原始技能包的对比」

---

## 一、触发条件

以下任一场景触发全量审计：

- [ ] 新增了理论课程/总纲文档（如吸收新课程01-26号）
- [ ] 用户说「检查有没有遗漏」「做个审计」「全面看看」
- [ ] 技能体系做了重大重组（如6月14日的模块化拆分）
- [ ] 连续校准发现超过3次「遗漏知识点」标记

---

## 二、审计流程（6步法）

### 第1步：盘清所有知识来源

```yaml
搜索范围：
  □ 课程原始素材（01-26号转写，含面授绝招/身强弱/性格/太岁等）
  □ 总纲文档（理论知识体系完整版/全量知识体系/七大维度二十五方面/终极总纲等）
  □ 知识提取报告（遗漏标记报告/二次扫描报告等）
  □ 参考文件（所有skill下的references/）
  □ 人物档案报告（家族成员+外部案例的命理分析）
  □ 脚本工具（bazi-engine.py等）
  □ 外部吸收（如bazi-destiny-analysis从外部Agent吸收的理论）

口诀：
  课程总纲参考档，人物案例和脚本
  七个来源都要搜，一个不落是底线
```

### 第2步：盘点所有技能的知识覆盖声明

```yaml
列出所有bazi技能及各自声明的覆盖领域：
  ☐ bazi-foundation-analysis → 总纲/能量倍数/天干地支/藏干/身强弱/神煞/纳音等
  ☐ bazi-education-analysis → 学业第0层/六步排查/文昌
  ☐ bazi-wealth-analysis → 财星评分/财富量化/财库
  ☐ bazi-career-analysis → 事业六维/格局/行业/名望
  ☐ bazi-misfortune-analysis → 灾祸/疾病/官司
  ☐ bazi-marriage-analysis → 婚姻/配偶/六亲
  ☐ bazi-children-analysis → 子女/生育
  ☐ bazi-liunian-analysis → 流年大运/刑冲合害
  ☐ bazi-health-psychology → 健康/性格
  ☐ bazi-fortune-analysis → 十神详解/补运/姓名学
  ☐ bazi-calibration → 校准原则/职业道德/审计
  ☐ 其余 → 验证/引擎/模板/主控

⚠️ 注意：不要只读SKILL.md的声明，要实际检查文件内容确认覆盖
```

### 第3步：逐层对比 → 分类（✅/⚠️/❌）

```yaml
对每个知识点，判断覆盖状态：

✅ 已覆盖 = 技能中有明确对应的章节/规则
  例：「文昌贵人查法」→ ref-shensha有完整查法

⚠️ 部分覆盖 = 有提及但不完整/分散未整合
  例：「性格分析」→ 十神性格表有，但五行性格/天干一气性格未整合

❌ 完全遗漏 = 任何技能中均未提及
  例：「社会名望分析」→ 任何技能都未覆盖

分类原则：
  - 宁可标⚠️不要标✅（严格标准）
  - 「有提到但不够详细」= ⚠️ 不是 ✅
  - 「理论上可能被推导出来但没写过」= ❌ 不是 ✅
```

### 第4步：冲突检查

```yaml
高冲突风险区域（必须逐项对比）：

  □ 能量倍数表：
    foundation vs liunian-analysis → 必须完全一致
    三会20/10倍，三合15/7倍，半三合10/5倍等

  □ 藏干标准：
    foundation vs 所有引用藏干的技能 → 必须统一100/60/30
    本气=12分，中气=7.2分，余气=3.6分

  □ 身强弱标准：
    foundation vs fortune-analysis vs 各子技能 → 必须统一
    <40分身弱，40-60分中和，>60分身强

  □ 大运排法：
    阳男阴女顺排，阴男阳女逆排

  □ 空亡规则：
    空亡=能量减半

口诀：
  能量倍数藏干表，身强弱和大运法
  空亡墓库十神论，逐项对比零冲突
```

### 第5步：补入遗漏

```yaml
按优先级补入：

🔴 高优先级（立即补）：
  - 影响断事核心维度的遗漏（如社会名望判定）
  - 用户明确指出的遗漏

🟡 中优先级（本批次补）：
  - 实用但非核心的遗漏（如补充神煞/建筑意象）
  - 已有基础但需增强整合的模块（如性格分析）

🟢 低优先级（可后续补）：
  - 细小规则/边缘知识（如养狗吉凶/称骨算命）
  - 不常用但记录了完整的内容

补入方式：
  ☐ 补入现有对应skill（优先）
  ☐ 创建新的reference文件（如内容较独立）
  ☐ 更新主控agent路由（如新增了skill或重要section）

并行补入策略（大批次时）：
  ☐ 按skill分组，每组用一个subagent处理
  ☐ 全部补完后统一做冲突检查
  ☐ 更新master agent路由表
```

### 第6步：输出审计报告

```yaml
报告必须包含：
  □ 审计范围（哪些来源 vs 哪些技能）
  □ 总体覆盖度（百分比 + ✅/⚠️/❌ 数量）
  □ 所有遗漏清单（按优先级排序）
  □ 已补入的项目和位置
  □ 冲突检查结果（数量+说明）
  □ 剩余待做（低优先级未补的）

报告格式示例：
  总知识量：~210项
  覆盖度：94%（198/210）
  冲突：0项
  高优：3/3 已补入 ✅
  中优：6/6 已补入 ✅
  低优：8项待补
```

## 三、实战案例：2026-06-14全量审计

**审计范围**：6份总纲文档 + 26课原始素材 + 3份遗漏标记报告 vs 17个bazi技能

**结果**：
| 指标 | 数值 |
|:-----|:----:|
| 总知识量 | ~210项 |
| 覆盖度 | 94%（↑6% from 88%）|
| 冲突 | 0项 |
| 补入 | 9项（3高+6中）|
| 剩余 | 8项低优 |

**关键经验**：
1. 用户纠正「审计不是新技能vs旧技能，是所有技能vs全部理论」→ 已编入本方法论
2. 并行补入效率高（用delegate_task同时处理6个skill的patch）
3. 完成后必须做冲突检查（本次0冲突，因所有补入内容在各自领域互不重叠）

## 四、常见陷阱

```yaml
① 只对比了部分来源 → ❌
   正确：必须搜全所有知识来源（课程+总纲+参考+报告+脚本）
   
② 只读了SKILL.md的声明没检查内容 → ❌
   正确：要打开文件确认对应的section确实存在
   
③ 审计完不补入 → ❌
   正确：审计的目的是找出遗漏并补入，不是纯做统计
   
④ 补入后不更新路由 → ❌
   正确：新增了重要章节后，更新master agent的路由表
   
⑤ 认为「拆分后的技能对比原始技能包」就是审计 → ❌
   用户明确说：审计是「所有技能 vs 全部理论」
   拆分后的技能和原始技能包其实是同一套东西的不同组织方式
```
