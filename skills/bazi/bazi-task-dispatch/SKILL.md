# 八字排盘 · 子任务分发引擎

> **来源**: 基于SOUL.md Sub-Agent分工体系 + HERMES.md 任务派发流程
> **应用场景**: 需要并发执行多个子任务时（引擎开发/前端开发/报告生成/测试验证等）

## §1. 核心原则

### 1.1 任务分发三原则

| 原则 | 内容 | 违反后果 |
|:----|:-----|:---------|
| **原则A** | 每个子任务必须有可脚本验证的终止条件 | 子Agent不知道什么时候算完成 |
| **原则B** | 四要素传递：目标+标准+格式+流程 | 子Agent靠猜测工作 |
| **原则C** | Maker/Checker分离：写的人不审，审的人不写 | 认知偏差导致bug遗漏 |

### 1.2 四要素传递模板

```yaml
delegate_task(
  goal="子任务具体目标",
  context=|
    🎯 目标: 具体要做什么
    📏 标准: 质量要求/规则约束（如"所有结论必须有引擎JSON行号"）
    📋 格式: 输入输出格式（如"返回JSON，包含x字段"）
    🔄 流程: 执行步骤顺序（如"①加载技能②取JSON③写分析"）
  |
)
```

### 1.3 不能委派的场景

| 场景 | 原因 | 替代方案 |
|:-----|:-----|:---------|
| 需要用户交互/确认 | 子Agent没有clarify工具 | 主Agent先问用户再委派 |
| 单步简单操作 | 不值得开一个子Agent | 直接用工具调用 |
| 需要总览全局一致性 | 子Agent没有上下文 | 主Agent做汇总 |
| 对速度敏感 | 子Agent启动有开销 | 直接用terminal执行 |

## §2. Sub-Agent角色定义

### 2.1 7个标准化角色

| 角色 | 职责 | 适用技能 | 核查项 |
|:----|:-----|:---------|:-------|
| **引擎开发员** | Python规则引擎实现 | addy-incremental-impl, addy-tdd | 单元测试通过率 |
| **前端开发员** | 前端SPA/小程序 | addy-frontend-ui | 移动端+21§顺序 |
| **API开发员** | FastAPI后端 | addy-api-design, addy-security | 契约完整性 |
| **测试验证员** | 测试编写+执行 | addy-tdd, bazi-auto-verify | 320条门禁 |
| **审核员** | Maker输出对抗审查 | addy-code-review | 五轴检查 |
| **命理分析师** | 八字命理推理 | bazi-foundation-analysis | 规则行号引用 |
| **报告生成员** | 21§报告格式化 | bazi-report-template | 引擎JSON一致性 |

### 2.2 角色选择矩阵

```
IF 任务涉及:
  引擎代码修改 → 引擎开发员
  前端页面修改 → 前端开发员
  API端点修改 → API开发员
  测试用例编写 → 测试验证员
  代码审查 → 审核员
  八字分析推理 → 命理分析师
  报告格式化输出 → 报告生成员
  多个独立子任务 → 按类型分别委派(并行)
```

## §3. 任务分解策略

### 3.1 拆解粒度标准

```
一个子任务 = 5-15次工具调用, 2-5分钟完成

过细（<5次调用）: 不值得开子Agent的开销
过粗（>20次调用）: 子Agent上下文会迷失

检查标准:
  □ 该子任务是否有明确的输入和输出？
  □ 能否用一句话描述任务内容？
  □ 是否有可脚本验证的完成条件？
  □ 是否需要调用多种不同类型工具？
```

### 3.2 常见八字排盘任务分解

#### 排盘+分析任务
```
task_1: 排盘 + 引擎评分 (工具调用密集型)
  角色: 命理分析师
  完成条件: /tmp/{姓名}_engine.json 存在且包含21§字段

task_2: 格局/喜用神分析 (推理密集型)
  角色: 命理分析师
  完成条件: 格局判定7步检查清单全部完成

task_3: 21§报告生成 (格式化输出)
  角色: 报告生成员
  完成条件: 21§全部填写完毕, 输出到/tmp/{姓名}_报告.md

task_4: 发布前校验 (验证密集型)
  角色: 测试验证员
  完成条件: 320条门禁全部通过
```

#### 引擎开发任务
```
task_1: 单元测试先行 (TDD)
  角色: 引擎开发员
  完成条件: 测试文件包含正向+负向+边界用例

task_2: 规则引擎实现 (最小实现)
  角色: 引擎开发员
  完成条件: 所有测试通过

task_3: 代码审查 (对抗性)
  角色: 审核员
  完成条件: 五轴审查全部通过
```

## §4. 委派执行流程

### 4.1 标准委派流程

```yaml
① 主Agent接收用户指令
② 分析是否需要拆分为子任务
  ├─ 是 → 按§3分解策略拆分
  │      每个子任务：目标+标准+格式+流程
  └─ 否 → 直接在main中完成
③ 确定子任务依赖关系
  ├─ 无依赖 → 并行委派（delegate_task tasks=[]）
  └─ 有依赖 → 串行委派（一个完成后再派下一个）
④ 调用delegate_task
⑤ 子Agent返回结果
⑥ 主Agent审核汇总（Maker/Checker原则）
  ├─ 通过 → 合并结果
  └─ 不通过 → 记录问题并重新委派修复
⑦ 向用户汇报最终结果
```

### 4.2 结果验证清单

子Agent返回后，主Agent必须验证以下内容：

```
□ 子Agent是否按给定的终止条件完成？
□ 子Agent的self-report是否可信？（铁律：sub-agent自述不可信）
□ 是否有实际的文件/数据产出？（不是"写好了"而是能查到bytes_written>0）
□ Maker/Checker是否分离？（写的人不是审的人）
□ 输出格式是否符合四要素中的格式要求？
```

### 4.3 常见失败模式

| 模式 | 表现 | 预防 |
|:-----|:-----|:-----|
| **幻觉式完成** | 子Agent说写好了但文件不存在 | 要求返回"exit_code=0"或"bytes_written>0" |
| **跑题** | 子Agent做了额外的事情 | 四要素中明确scope边界 |
| **超时** | 子Agent任务过大 | 控制子任务在5分钟以内 |
| **自洽但不正确** | 自圆其说但不符合原始理论 | 明确要求"标注规则行号" |
| **语言污染** | 中英文混杂/语气不对 | context中指定语言风格 |

## §5. 并发与资源管理

### 5.1 并发限制

| 资源 | 限制 |
|:-----|:-----|
| 最大并行子Agent数 | 4个（delegation.max_concurrent_children） |
| 最大嵌套深度 | 1层（子Agent不能继续委派） |
| 单个子Agent超时 | 5分钟（300秒） |

### 5.2 资源规划

```
IF 任务包含:
  4个以下独立子任务 → 全部并行委派
  5-8个子任务 → 分2批（每批4个）
  依赖链 → 串行（A→B→C）
```

## §6. 应用场景示例

### 场景A: 全量审计
```
主任务: 审计所有bazi技能文件
分解:
  task_1: 检查技能文件存在性 (测试验证员)
  task_2: 检查引擎代码一致性 (审核员)
  task_3: 检查SOP引用完整性 (命理分析师)
执行: 无依赖 → 并行
汇总: 主Agent合并3份报告
```

### 场景B: 修复引擎Bug
```
主任务: 修复身强弱判定Bug
分解:
  task_1: 查原始理论确认规则 (命理分析师)
  task_2: 修改引擎代码 (引擎开发员)
  task_3: 测试验证 (测试验证员)
执行: task_1→task_2→task_3 串行
汇总: 主Agent验证测试结果
```

### 场景C: 出报告
```
主任务: 给某人生成八字报告
分解:
  task_1: 排盘+引擎评分 (命理分析师)
  task_2: 21§报告生成 (报告生成员)
  task_3: 发布前校验 (测试验证员)
执行: task_1→task_2→task_3 串行
汇总: 主Agent确认报告文件存在+门禁通过
```

## §7. 与SOP的集成

```yaml
SOP Phase 0-3: 主Agent执行（不委派）
  系统就绪 → 加载技能 → 排盘 → 引擎评分

SOP Phase 4 (分析+出报告):
  └─ 可委派给「命理分析师」执行部分§的分析
  └─ 可委派给「报告生成员」执行§格式化输出
  └─ 主Agent必须做Maker/Checker

SOP Phase 5 (发布前校验):
  └─ 可委派给「测试验证员」跑320条门禁
  └─ 主Agent验证结果

SOP Phase 6-7 (推库+纠错):
  └─ 主Agent执行（不委派）
```
