# 八字排盘 · 子任务分发引擎

> **来源**: 基于SOUL.md Sub-Agent分工体系 + HERMES.md 任务派发流程
> **应用场景**: 需要并发执行多个子任务时（引擎开发/前端开发/报告生成/测试验证等）

## §1. 核心原则

### 1.1 任务分发三原则

| 原则 | 内容 | 违反后果 |
|:----|:-----|:---------|
| **原则A** | 每个子任务必须有可脚本验证的终止条件 | 子Agent不知道什么时候算完成 |
| **原则B** | 四要素传递：目标+标准+格式+流程 | 子Agent靠猜测工作 |
| **原则C** | Maker/Checker分离：写的人不审，审的人不写 | 认知偏差导致bug遗漏 |

### 1.2 四要素传递模板

```yaml
delegate_task(
  goal="子任务具体目标",
  context=|
    🎯 目标: 具体要做什么
    📏 标准: 质量要求/规则约束（如"所有结论必须有引擎JSON行号"）
    📋 格式: 输入输出格式（如"返回JSON，包含x字段"）
    🔄 流程: 执行步骤顺序（如"①加载技能②取JSON③写分析"）
  |
)
```

### 1.3 不能委派的场景

| 场景 | 原因 | 替代方案 |
|:-----|:-----|:---------|
| 需要用户交互/确认 | 子Agent没有clarify工具 | 主Agent先问用户再委派 |
| 单步简单操作 | 不值得开一个子Agent | 直接用工具调用 |
| 需要总览全局一致性 | 子Agent没有上下文 | 主Agent做汇总 |
| 对速度敏感 | 子Agent启动有开销 | 直接用terminal执行 |

## §2. Sub-Agent角色定义

### 2.1 7个标准化角色

| 角色 | 职责 | 适用技能 | 核查项 |
|:----|:-----|:---------|:-------|
| **引擎开发员** | Python规则引擎实现 | addy-incremental-impl, addy-tdd | 单元测试通过率 |
| **前端开发员** | 前端SPA/小程序 | addy-frontend-ui | 移动端+21§顺序 |
| **API开发员** | FastAPI后端 | addy-api-design, addy-security | 契约完整性 |
| **测试验证员** | 测试编写+执行 | addy-tdd, bazi-auto-verify | 320条门禁 |
| **审核员** | Maker输出对抗审查 | addy-code-review | 五轴检查 |
| **命理分析师** | 八字命理推理 | bazi-foundation-analysis | 规则行号引用 |
| **报告生成员** | 21§报告格式化 | bazi-report-template | 引擎JSON一致性 |

### 2.2 角色选择矩阵

```
IF 任务涉及:
  引擎代码修改 → 引擎开发员
  前端页面修改 → 前端开发员
  API端点修改 → API开发员
  测试用例编写 → 测试验证员
  代码审查 → 审核员
  八字分析推理 → 命理分析师
  报告格式化输出 → 报告生成员
  多个独立子任务 → 按类型分别委派(并行)
```

## §3. 任务分解策略

### 3.1 拆解粒度标准

```
一个子任务 = 5-15次工具调用, 2-5分钟完成

过细（<5次调用）: 不值得开子Agent的开销
过粗（>20次调用）: 子Agent上下文会迷失

检查标准:
  □ 该子任务是否有明确的输入和输出？
  □ 能否用一句话描述任务内容？
  □ 是否有可脚本验证的完成条件？
  □ 是否需要调用多种不同类型工具？
```

### 3.2 常见八字排盘任务分解

#### 排盘+分析任务
```
task_1: 排盘 + 引擎评分 (工具调用密集型)
  角色: 命理分析师
  完成条件: /tmp/{姓名}_engine.json 存在且包含21§字段

task_2: 格局/喜用神分析 (推理密集型)
  角色: 命理分析师
  完成条件: 格局判定7步检查清单全部完成

task_3: 21§报告生成 (格式化输出)
  角色: 报告生成员
  完成条件: 21§全部填写完毕, 输出到/tmp/{姓名}_报告.md

task_4: 发布前校验 (验证密集型)
  角色: 测试验证员
  完成条件: 320条门禁全部通过
```

#### 引擎开发任务
```
task_1: 单元测试先行 (TDD)
  角色: 引擎开发员
  完成条件: 测试文件包含正向+负向+边界用例

task_2: 规则引擎实现 (最小实现)
  角色: 引擎开发员
  完成条件: 所有测试通过

task_3: 代码审查 (对抗性)
  角色: 审核员
  完成条件: 五轴审查全部通过
```

## §4. 委派执行流程

### 4.1 标准委派流程

```yaml
① 主Agent接收用户指令
② 分析是否需要拆分为子任务
  ├─ 是 → 按§3分解策略拆分
  │      每个子任务：目标+标准+格式+流程
  └─ 否 → 直接在main中完成
③ 确定子任务依赖关系
  ├─ 无依赖 → 并行委派（delegate_task tasks=[]）
  └─ 有依赖 → 串行委派（一个完成后再派下一个）
④ 调用delegate_task
⑤ 🚨 子Agent返回后，**必须物理验证**文件是否真正落地
  ├─ 用 terminal('ls -la <路径>/<文件>') 确认文件存在且>0字节
  └─ 子Agent自述"写好了"不可信，必须自己查一次
⑥ 主Agent审核汇总（Maker/Checker原则）
  ├─ 通过 → 合并结果
  └─ 不通过 → 记录问题并重新委派修复
⑦ 向用户汇报最终结果
```

### 4.2 结果验证清单

子Agent返回后，主Agent必须物理验证以下内容：

```
□ 📁 文件存在性: terminal('ls -la <文件路径>') → bytes > 0
   （子Agent说"写好了"不算，物理查到才算）
□ 💬 文件内容匹配: terminal('head -5 <文件路径>') → 内容正确
□ ✅ 终止条件完成: 子Agent声称的完成项是否与要求一致
□ 🔍 标题/编号匹配: §顺序/文件名编号是否符合模板规范
□ 🎨 格式合规: 九龙道长版§1~§21顺序对不对？
                 盲派版体用定义是否正确（体含食伤用只财官）？
□ 📏 Maker/Checker: 写的人不是审的人
```

### 4.3 常见失败模式

| 模式 | 表现 | 预防 |
|:-----|:-----|:-----|
| **幻觉式完成** | 子Agent说写好了但文件不存在 | 🚨 物理验证：terminal('ls') 查文件 |
| **跑题** | 子Agent做了额外的事情 | 四要素中明确scope边界 |
| **超时** | 子Agent任务过大 | 控制子任务在5分钟以内 |
| **自洽但不正确** | 自圆其说但不符合原始理论 | 明确要求"标注规则行号" |
| **格式错误** | §顺序乱/体用定义错误 | context固化模板+体用定义 |
| **语言污染** | 中英文混杂/语气不对 | context中指定语言风格 |

### 4.4 🚨 铁律（2026-07-16 主母+立谎报完成触发）

```yaml
子Agent返回status=completed 后，主Agent必须立即执行：
  
  ✅ 第1步 — 物理查文件
     terminal('ls -la <输出路径>/<文件>')
     期望: 文件存在，size > 0
     若不存在 → 子Agent谎报，重新派发

  ✅ 第2步 — 查git状态
     cd /root/weiwuji-knowledge-base && git status --short
     子Agent可能没commit → 主Agent手动commit

  ✅ 第3步 — 验文件内容
     terminal('head -5 <文件>') 确认内容正确非占位符
```

**教训**：2026-07-16 派发主母+立的盲派v2报告，子Agent返回"写好了"status=completed，实际文件不存在。根因：子Agent的self-report不可信，必须物理验证。

### 4.3 常见失败模式

| 模式 | 表现 | 预防 |
|:-----|:-----|:-----|

## §5. 并发与资源管理

### 5.1 并发限制

| 资源 | 限制 |
|:-----|:-----|
| 最大并行子Agent数 | 4个（delegation.max_concurrent_children） |
| 最大嵌套深度 | 1层（子Agent不能继续委派） |
| 单个子Agent超时 | 5分钟（300秒） |

### 5.2 资源规划

```
IF 任务包含:
  4个以下独立子任务 → 全部并行委派
  5-8个子任务 → 分2批（每批4个）
  依赖链 → 串行（A→B→C）
```

## §6. 应用场景示例

## 8.6 大运真假判定规则

大运天干在原局的「有」= 以下任一：
  ├─ 天干有明字 → 有 ✅
  ├─ 天干无明字，但地支藏干**本气/中气/余气**有该字 → 有 ✅
  └─ 天干无明字，地支也无该藏干 → 无 ❌

**坑**：藏干余气(10-30%)也算「有」，不能因比例低就判无。
详见 `bazi-foundation-analysis` 参考文件 `大运真假判定精确规则_20260716.md`。

## §9. 与SOP的集成

**背景**：文颖报告用了错误§格式，盲派报告用了错误体用定义。根因：sub-agent context缺少格式锁定。

### 8.1 九龙道长版 context 必含
```
标准: |
  🚨 ① 先 skill_view('bazi-report-template') 再写报告
  🚨 ② 严格按§1~§21顺序（§1八字排盘→...→§21总结）
  🚨 ③ 盲派体用：体=日主+印+比劫+食伤，用=财+官
  🚨 ④ 穷举清单扫所有位置藏干
  🚨 ⑤ 写前 touch /tmp/.bazi_verified
```

### 8.2 盲派版 context 必含
```
标准: |
  🚨 ① 体=日主+印+比劫+食伤(食神+伤官全算体)
       用=财+官(正财偏财正官七杀，只有这两个)
       做功=体→追求→用的方式
  🚨 ② 15步盲派结构：阴阳→五行→体用→格局→做功→财官→
       十神→大运→扛力→象法→财富→事业→婚姻→健康→总结
  🚨 ③ 每算体用前穷举所有位置
  🚨 ④ 写前 touch /tmp/.bazi_verified
```

## 8.4 口诀引用前条件校验（2026-07-16 四柱皆带财bug触发）

> 子代理引用口诀「四柱皆带财，库在日时来，食伤能生财，不富没天理」，未核对年柱是否有财。
> 家主八字年柱庚申(劫财)无财 → 口诀条件不满足 → 不应直接引用

**铁律**：引用任何口诀/断语前，必须逐条核对条件是否满足本八字。

```yaml
核查流程：
  □ Step 1 — 通读口诀的每个条件
  □ Step 2 — 逐条件核对八字数据（四柱每个字都要查）
  □ Step 3 — 条件全满足 → 可引用
  □ Step 4 — 任一条件不满足 → 禁止引用！须注明差异
  □ Step 5 — 有条件满足但不全面的 → 在引用后注明具体差异

常见陷阱口诀：
  四柱皆带财(查年柱有无财)→缺则改"三柱有财"
  身弱不担财(核实身弱与否)→核实身弱数据
  四柱纯阳(查各柱阴阳)→核实阴阳
  杀印相生必贵(查印是否有根)→核实印根
  
  🚨 泉师兄口诀/断语引用前，条件不满足→不准引用，须注明差异
```

## §7. 与SOP的集成

```yaml
SOP Phase 0-3: 主Agent执行（不委派）
  系统就绪 → 加载技能 → 排盘 → 引擎评分

  └─ 主Agent执行（不委派）
```
