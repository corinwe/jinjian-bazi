# 物理化八字分析管线 · 设计方案

## 一、当前问题诊断

```
现状：人脑驱动 → 凭记忆写 → 跳步 → 出错 → 老板指出 → 再改
                                                  ↑
                                          每次都回到这里
```

**根因：没有物理约束。** 我「可以」跳过Step5、「可以」不列规则清单、「可以」凭感觉写生克关系。没有任何机制阻止我犯错。

## 二、物理化方案

### 核心思路：把流程封装成 shell 脚本 + Python 校验

```
入口命令（一个命令跑完整个管线）
  │
  ├─ Step 1-3：自动排盘 + 引擎评分 → 输出 engine_data.json
  ├─ Step 4：自动加载所有理论技能 → 输出 rule_checklist_{§}.md
  ├─ Step 5：强制填规则vs数据对照表（不填不能继续）
  ├─ Step 6：写分析内容（人写，但框架已定）
  ├─ Step 7：自动校验 → verify_report.py
  └─ Step 8：推库
```

### Step 1-3：一键跑引擎 → engine_data.json

```bash
# 输入八字 → 输出引擎JSON
cd /root/bazi-platform/scripts
bash bazi-analyze-prep.sh \
  --name "魏启令" --gender 男 \
  --year 1980 --month 8 --day 6 --hour 6
```

**输出：`/tmp/bazi-engine-data/{姓名}/engine_data.json`**

这个JSON包含了所有后续需要的数据：
- paipan（四柱干支）
- sec_1_overview（纳音/空亡/起运/ri_zhu）
- sec_3_shen_qiang_ruo（身强弱评分明细）
- sec_8_wealth（财星评分明细）
- sec_17_da_yun_detail（大运列表）
- 等等

### Step 4：自动生成规则对照表模板

```bash
# 为每个§生成对照表模板
python3 /root/bazi-platform/scripts/gen_rule_checklist.py \
  --engine /tmp/bazi-engine-data/魏启令/engine_data.json \
  --section sec_3_shen_qiang_ruo \
  --output /tmp/checklist/sec_3.md
```

**输出样板（自动填充数据，留空结论）：**

```markdown
# §3 身强弱 规则对照表

| # | 规则（来自foundation-analysis.skill） | 命主数据（引擎） | 结论（人填） |
|:-:|:--------------------------------------|:-----------------|:------------|
| 1 | 月令本气印=40分，中气/余气印=0分    | 月令未，本气己土=偏印 | 40分 ✅ |
| 2 | 天干比劫全计分                       | 年干庚(金)+时干辛(金) | 20分 ✅ |
| 3 | 燥土：壬/癸灭火→当土看计分          | 月干癸水灭火         | 当土看 ✅ |
| 4 | 从弱格：0分→恒定50分                | 总分64>0，有印比     | 非从弱 ✅ |
```

**关键：自动填充「命主数据」列，我只填「结论」列。** 
这样我就不能跳过数据检查——数据已经摆在那里了，我只需要判断对不对。

### Step 5：强制填对照表（物理闸门）

```bash
# 检查对照表是否全部填完
python3 /root/bazi-platform/scripts/check_checklist.py \
  --checklist /tmp/checklist/sec_3.md
# 如果有未填的行 → 报错退出，不让我继续写分析
```

**效果：我「必须」逐条过规则→数据的匹配，不能跳步。**

### Step 6：写分析（框架已定）

对照表填完并通过后，我才开始写分析段落。
这时候框架已经有了——规则是什么、数据是什么、结论是什么——我只需要润色成自然语言。

### Step 7：自动校验

```bash
python3 /root/bazi-platform/scripts/verify_report.py \
  --report 报告文件.md \
  --engine /tmp/bazi-engine-data/魏启令/engine_data.json
```

**校验项（物理不可跳过）：**

| 校验项 | 检查方法 | 发现的问题类型 |
|:-------|:---------|:---------------|
| 身强弱分数一致 | grep报告中的分数 vs engine.json | 写错数字 |
| 财星分数一致 | 同上 | 写错数字 |
| 五行生克方向 | 检查「金生土」「土生金」等关键词 | 生克写反 |
| 十神名称正确 | 检查「正印」「偏印」「正财」「偏财」 | 十神搞混 |
| 大运年份正确 | 检查大运起始年份 vs engine.json | 年份错误 |
| 喜忌一致 | 报告喜忌 vs 引擎喜忌 | 喜忌写反 |
| 空亡正确 | 报告中空亡 vs 引擎空亡 | 空亡算错 |

**不通过 → 阻断，修正后才能推库。**

### Step 8：推库（最后一步）

校验通过后，自动执行：
```bash
cd /root/weiwuji-knowledge-base && git add -A && git commit -m "..." && git push
```

## 三、每§的固定产出物

每个§在最终报告中对应一段分析，但我的**内部工作产物**应该是三段：

```
§X 分析
  ├─ checklist/§X.md     → 规则vs数据对照表（物理化Step5）
  ├─ draft/§X.md         → 分析草稿（人写）
  └─ 最终报告中的对应段落
```

**checklist是必须的，不生成checklist就不能写分析。** 这就像工程师画电路图之前必须列物料清单一样——不是可选项，是前置条件。

## 四、物理闸门汇总

| 闸门 | 位置 | 检查内容 | 不通过后果 |
|:----|:-----|:---------|:-----------|
| ① checklist完整性 | Step5→Step6之间 | 对照表所有行已填 | ⛔ 不给写分析 |
| ② 生克方向检查 | Step6→Step7之间 | 五行生克关系正确 | ⛔ 不给推库 |
| ③ 十神名称检查 | Step6→Step7之间 | 正印/偏印等正确 | ⛔ 不给推库 |
| ④ 数据一致性 | Step7 | 报告vs引擎全部一致 | ⛔ 不给推库 |
| ⑤ PIPELINE-SIG | 推库前 | 文件末尾有签名 | ⛔ 不给推库 |

## 五、「物理化」vs「随意发挥」的对比

| 维度 | 现在（随意发挥） | 改造后（物理化） |
|:-----|:-----------------|:-----------------|
| 规则来源 | 我脑子里记了多少用多少 | 从skill文件读取，一条不能少 |
| 数据核对 | 凭印象核对 | checklist自动填充引擎数据 |
| 生克关系 | 凭感觉写 | 有固定方向检查表 |
| 校验 | 无 | verify_report.py 自动跑 |
| 跳过步骤 | 可以 | 物理闸门阻断，跳不过 |
| 可重复性 | 每次看状态 | 每次流程一致 |
